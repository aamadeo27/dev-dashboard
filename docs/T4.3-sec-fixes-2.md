# T4.3-sec-fixes-2 — Security Fix Pass 2

## What was done

Applied four security fixes (FIX-SEC-5 through FIX-SEC-8) to the Tauri 2 backend:

- **FIX-SEC-5**: Reverted `launch_run` spawn ordering back to insert-before-spawn.
- **FIX-SEC-6**: Strengthened `is_valid_sequence_name` to reject `..` segments, leading `/`, and names over 256 chars.
- **FIX-SEC-7**: Added lexical `..` normalization inside `verify_run_dir_prefix` before the prefix check.
- **FIX-SEC-8**: Added length caps and control-character stripping to `log_frontend_error` inputs.

## How the affected components work

### FIX-SEC-5 — insert-before-spawn ordering (`launch_run`)

The sessions map stores an `Arc<SessionHandle>` keyed by `run_id`. `run_io_loop` removes the entry (Step 8) when the child exits. If the child exits before the map insert runs, the remove is a no-op and the handle is never inserted — clean. The reverse (insert-after-spawn as FIX-SEC-4 had it) creates a window where `run_io_loop`'s `map.remove` fires before `map.insert`, leaving a stale entry that never gets cleaned up.

The stop_run concern FIX-SEC-4 tried to solve is not real: `CancellationToken::cancel()` is idempotent and the I/O task's first `select!` branch detects the already-cancelled token immediately on its first poll.

### FIX-SEC-6 — `is_valid_sequence_name` path-traversal guards

The function now gates on three additional conditions before the character allowlist check:
1. `name.len() > 256` — rejects excessively long names.
2. `name.starts_with('/')` — rejects absolute paths (e.g. `/etc/passwd`).
3. `name.split('/').any(|seg| seg == "..")` — rejects any segment equal to `..` (e.g. `../../etc/passwd`, `../sibling`, `sub/../etc`).

The character allowlist (`[A-Za-z0-9._\-/ ]`) already prevented most injection, but `..` is composed entirely of allowed chars (`.`), so it needed an explicit segment check.

### FIX-SEC-7 — lexical `run_dir` normalization (`verify_run_dir_prefix`)

A new private helper `normalize_path_lexically` walks forward-slash-split path components:
- Skips empty components and `.`
- Pops the last accumulated segment on `..`
- Pushes everything else

This is applied to `run_dir` (after `\\` → `/` normalisation) before the `starts_with` prefix check. Without this, a path like `<project>/.claude/runs/../../outside` would pass the prefix check because the raw string starts with the expected prefix before the `../..` escape.

The project path is still fully canonicalized via `tokio::fs::canonicalize` (resolves symlinks, OS-level traversal). The lexical normalization is a defense-in-depth layer for the `run_dir` argument which may not yet exist on disk.

### FIX-SEC-8 — `log_frontend_error` input sanitization

An inline closure `sanitize(s, max_len)` is applied to all three input fields before logging:
- Filters characters: keeps `\n`, `\t`, printable ASCII (`\x20`–`\x7E`), rejects `\x7F` (DEL) and all other control chars (`\x00`–`\x1F` except `\n`/`\t`).
- Takes only `max_len` chars after filtering (`message` 4096, `stack` 8192, `route` 512).

The `unicode chars()` iterator means multi-byte UTF-8 sequences are handled correctly — each char is tested atomically. The correlation_id, structured log fields, and log level are unchanged.

## Decisions made within Task scope

- `normalize_path_lexically` is a private `fn` (not `pub`) — it is only needed by `verify_run_dir_prefix` in the same file.
- The `sanitize` closure in `log_frontend_error` is inline (not a named function) per the task spec.
- The `..` check in `is_valid_sequence_name` splits on `/` only (not `\`), consistent with the function's existing allowlist which does not include `\`.

### Manual spot-checks

- `is_valid_sequence_name("../../etc/passwd")` → `false`
- `is_valid_sequence_name("/absolute")` → `false`
- `is_valid_sequence_name(&"a".repeat(257))` → `false`
- `is_valid_sequence_name(&"a".repeat(256))` → `true`
- `verify_run_dir_prefix(project, project/.claude/runs/../../outside)` → `PermissionDenied`
