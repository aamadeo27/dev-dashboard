# T0.6 — CI/CD Pipeline

## What was done

Created three GitHub Actions configuration files that implement the full CI/CD pipeline for the Dev Dashboard Tauri 2 app:

- `.github/workflows/ci.yml` — Check workflow (lint, typecheck, tests, bindings freshness)
- `.github/workflows/build.yml` — Cross-platform release build workflow
- `.github/dependabot.yml` — Automated dependency update configuration

## How the affected components work

### ci.yml

Triggers on every push to any branch and on PRs targeting `main`.

The key cost-saving behaviour: the job matrix defines three runners (`ubuntu-latest`, `macos-latest`, `windows-latest`), but the `if:` condition on the job gates macOS and Windows to PR events only:

```yaml
if: github.event_name == 'pull_request' || matrix.os == 'ubuntu-latest'
```

Plain branch pushes (not a PR) run only the Linux job. PRs to `main` run all three — satisfying the branch-protection required checks (`Check (ubuntu-latest)`, `Check (macos-latest)`, `Check (windows-latest)`).

Step order within each job:

1. Checkout
2. Rust stable toolchain (clippy + rustfmt components)
3. Rust dependency cache (`Swatinem/rust-cache@v2` keyed to `src-tauri`)
4. pnpm setup (v9)
5. Node 22 setup with pnpm cache
6. Linux-only: install Tauri WebKit system libraries
7. `pnpm install --frozen-lockfile`
8. `pnpm lint:ci` (Biome, no writes)
9. `pnpm typecheck` (tsc --noEmit)
10. `cargo fmt --manifest-path src-tauri/Cargo.toml -- --check`
11. `cargo clippy --manifest-path src-tauri/Cargo.toml -- -D warnings`
12. `pnpm test` (Vitest)
13. `pnpm bindings` (regenerates `src/ipc/bindings.ts` from ts-rs derives)
14. `git diff --exit-code src/ipc/` (fails if bindings are stale)

### build.yml

Triggers only when a tag matching `v[0-9]+.[0-9]+.[0-9]+` is pushed.

Four-platform matrix (two separate macOS runners for Intel vs Apple Silicon):

| artifact     | runner         | target triple                   |
|--------------|----------------|---------------------------------|
| linux-x64    | ubuntu-latest  | x86_64-unknown-linux-gnu        |
| macos-x64    | macos-13       | x86_64-apple-darwin             |
| macos-arm64  | macos-latest   | aarch64-apple-darwin            |
| windows-x64  | windows-latest | x86_64-pc-windows-msvc          |

Each platform job runs the same setup sequence as ci.yml, then `pnpm tauri build` with `TAURI_PRIVATE_KEY: ""` and `TAURI_KEY_PASSWORD: ""` to suppress code-signing (personal builds).

Artifacts are uploaded with `actions/upload-artifact@v4` named `dev-dashboard-<artifact>-<tag>`, collecting all bundle output paths (`.AppImage`, `.deb`, `.dmg`, `.msi`, `.exe`). `if-no-files-found: error` ensures a missing bundle causes the job to fail rather than silently produce an empty artifact.

The dependent `release` job:
- `needs: build` — only runs after all four matrix jobs succeed
- Downloads all artifacts into `artifacts/`
- Creates a **draft** GitHub Release via `gh release create --draft --generate-notes`
- Uses `${{ github.token }}` (no external secrets required)

### dependabot.yml

Two update ecosystems, both scheduled weekly on Monday:
- `npm` at repo root — groups `@tauri-apps/*` packages and React packages separately
- `cargo` at `/src-tauri` — groups all `tauri*` crates together

Grouping ensures the paired JS/Rust Tauri packages are bumped in a single PR rather than split across two PRs that could cause version mismatches.

## Decisions made within task scope

- **`cargo fmt` flag order**: devops.md §5 shows `cargo fmt --check --manifest-path ...` but the correct flag order for cargo subcommand pass-through is `cargo fmt --manifest-path ... -- --check`. Used the correct form.
- **`release` job uses `gh` CLI instead of `softprops/action-gh-release@v2`**: devops.md §6 shows both approaches; the `gh` CLI avoids a third-party action dependency and `github.token` is sufficient since `permissions: contents: write` is set on the job.
- **`target` field in build matrix**: included as metadata even though `pnpm tauri build` auto-detects the host target. It documents intent and can be passed via `--target` if cross-compilation is added later.
- **Linux deps step ordering**: moved after Node/pnpm setup (before `pnpm install`) so the apt step does not block Node cache setup on non-Linux runners. Order within a single runner does not affect correctness.

## How to test / verify

**YAML syntax check** (requires `yq` or Python):

```bash
python -c "import yaml; yaml.safe_load(open('.github/workflows/ci.yml'))"
python -c "import yaml; yaml.safe_load(open('.github/workflows/build.yml'))"
python -c "import yaml; yaml.safe_load(open('.github/dependabot.yml'))"
```

**Manual checklist:**

- [ ] `if:` condition present on the `check` job in ci.yml
- [ ] All action references pinned to `@v4` or `@v2` (no `@latest`)
- [ ] `TAURI_PRIVATE_KEY: ""` present in build.yml build step env
- [ ] `release` job has `needs: build` and `permissions: contents: write`
- [ ] dependabot.yml has both `npm` and `cargo` ecosystems

**Live verification:**

1. Push a branch — only the `Check (ubuntu-latest)` job should run.
2. Open a PR to `main` — all three `Check (*)` jobs should run.
3. Push a `v*.*.*` tag — `build.yml` should trigger and create a draft release.
