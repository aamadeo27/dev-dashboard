# T2.8 — open_in_editor and open_in_terminal Commands

## What was done

Implemented two Tauri IPC commands that open a registered project in an editor
or terminal. Both commands look up the project path from `ProjectRegistry` by id,
emit a `toast:show` error event on failure, and return `AppError::Io` on failure.
Added `TOAST_SHOW` event constant to `ipc/events.rs`. Wired TypeScript wrappers
and wrote unit tests.

## How the affected component works

### Data flow

1. Frontend calls `openInEditor(id)` or `openInTerminal(id)`.
2. Tauri routes to `ipc::commands::open_in_editor` / `open_in_terminal`.
3. Command acquires `AppState.projects` mutex lock, calls
   `platform::lookup_project_path(&registry, &id)` to resolve the path.
   - Returns `AppError::NotFound` immediately if id is absent.
   - Mutex is released before the OS open call.
4. `platform::open_in_editor_impl` or `platform::open_in_terminal_impl` is
   called with the resolved path and an `AppHandle` reference.
5. **Editor**: if `$EDITOR` env var is set, spawns `$EDITOR <path>` as a
   detached child (`kill_on_drop(false)`). Otherwise falls back to
   `tauri_plugin_opener::open_path(path, None)`.
6. **Terminal**: always uses `tauri_plugin_opener::open_path(path, None)`.
7. On any open failure: emits `toast:show` with `kind="error"` and returns
   `AppError::Io`. Both happen unconditionally (not either/or).
8. `project.is_missing = true` does not prevent the attempt — the OS handles
   the path.

### Key functions

- `platform::editor::lookup_project_path(registry, id)` — pure path-lookup
  helper; used by both commands. Extracted for testability.
- `platform::editor::open_in_editor_impl(id, path, app)` — async; spawns
  editor or calls opener.
- `platform::terminal::open_in_terminal_impl(id, path, app)` — async; calls
  opener.
- `ipc::commands::open_in_editor` / `open_in_terminal` — thin Tauri command
  shims that lock the registry, look up the path, and delegate.

### Event: toast:show

Payload: `{ kind: "error", title: "Cannot open editor"|"Cannot open terminal", body: <error string> }`.
Constant: `crate::ipc::events::TOAST_SHOW = "toast:show"`.
Emitted via `app.emit(TOAST_SHOW, serde_json::json!({...})).ok()` — `.ok()`
suppresses emit errors (not actionable).

## Decisions made within Task scope

- `lookup_project_path` was extracted from `editor.rs` (not duplicated in
  `terminal.rs`) and re-exported from `platform/mod.rs`. Both commands use it.
  This avoids duplicating the pattern while keeping the module split clean.
- The mutex is locked and released before calling `open_in_editor_impl` /
  `open_in_terminal_impl` so the OS open call does not hold the registry lock.
- `$EDITOR` spawning uses `kill_on_drop(false)` — the editor process is
  detached and fully owned by the OS, not the Tauri process.
- Rust unit tests are written only for `lookup_project_path` (pure, testable
  without a Tauri runtime). The `open_in_*_impl` functions require
  `tauri::AppHandle` and are covered only by manual smoke tests (same precedent
  as `open_logs_folder` in T1.5).

### Manual smoke tests (required for acceptance on all 3 OSes)

| Scenario | macOS | Windows | Linux |
|----------|-------|---------|-------|
| `$EDITOR` set to `code` (VS Code) — registered project — opens VS Code at project dir | required | required | required |
| `$EDITOR` not set — registered project — opens OS default for the directory | required | required | required |
| `open_in_editor` with unknown id — toast with `kind=error`, `title="Cannot open editor"`, returns NOT_FOUND | required | required | required |
| `open_in_terminal` — registered project — opens OS terminal at project dir | required | required | required |
| `open_in_terminal` with unknown id — toast with `kind=error`, `title="Cannot open terminal"`, returns NOT_FOUND | required | required | required |
| Both commands with `is_missing=true` project — OS error surfaces as toast + IO return | required | required | required |

**macOS note**: `tauri_plugin_opener::open_path` on a directory typically opens
Finder. Terminal integration may require a registered handler (e.g., iTerm2
"Open in Terminal" service) or a separate launcher command.

**Windows note**: `tauri_plugin_opener` uses `ShellExecuteW`; directory opens
Explorer by default. `$EDITOR` spawning tested with `code.cmd` (VS Code on
PATH).

**Linux note**: `tauri_plugin_opener` calls `xdg-open`; terminal integration
depends on the desktop environment's default handler for directories.

### Why open_in_editor_impl and open_in_terminal_impl cannot be unit-tested

Both `open_in_editor_impl` and `open_in_terminal_impl` accept a `&tauri::AppHandle` parameter.
`tauri::AppHandle` cannot be constructed outside a running Tauri runtime — there is no public
constructor or mock interface. Any test that calls these functions directly would require spinning
up a full Tauri application context, which is only possible in an end-to-end scenario (Tauri
test harness or WebDriver). This is the same precedent established in T1.5 for `open_logs_folder`.
The testable logic (path lookup) has been extracted into `lookup_project_path` and is covered
by the Rust unit tests above. OS-level open behavior is verified by manual smoke tests.

### Regression risks

- `CLI_LOST` constant in `src/ipc/events.ts` must not be accidentally changed (TS-G9 guards this).
- The `{ id }` argument shape is the same for `remove_project`; changes to argument serialization
  conventions could silently break both. The existing `removeProject` tests cover this peer function.
- `lookup_project_path` is shared by both commands. A bug in the helper breaks both `open_in_editor`
  and `open_in_terminal` simultaneously — the shared helper is the single point of failure for
  path resolution across both commands.
