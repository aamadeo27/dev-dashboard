# T2.1-fixes-2: ProjectRegistry Review Fixes (Round 2)

## What was done

Applied four targeted fixes to the `ProjectRegistry` code in response to code-review findings from iteration 2. No behavior changes beyond what is listed; all existing tests continue to pass.

## Fixes applied

### FIX-1 (MEDIUM): `list_projects` made async

`list_projects()` was synchronous and called `p.path.exists()` (blocking `std::fs` stat) on each project entry while a `tokio::sync::Mutex` was held. This was inconsistent with the `tokio::fs` pattern used everywhere else in the module.

The method is now `pub async fn list_projects(&self) -> Vec<Project>` and uses `tokio::fs::try_exists` to check path existence. All five test call sites in `mod.rs` and the one call site in `commands.rs` were updated to `.await` the result.

### FIX-2 (MEDIUM): `load()` doc explains intentional sync I/O

`load()` is the only synchronous constructor in an otherwise fully async module. A doc comment was added explaining that `std::fs` is used intentionally because `load()` is called once at startup before the Tokio runtime is initialised.

### FIX-3 (LOW): `rename_project` doc corrected

The previous doc comment said "internal only; not exposed via TS wrappers per KB §5.1". Changed to the precise formulation: `/// Not exported as a TypeScript binding per KB §5.1; debug-build IPC only.`

### FIX-4 (LOW): `canonicalize_dir` comment corrected

The comment claimed `tokio::fs::canonicalize` verifies existence "in one async call", which was misleading because a second `metadata` stat immediately follows. Changed to: `// tokio::fs::canonicalize resolves symlinks and verifies existence.`

## How the affected component works

`ProjectRegistry` is an in-memory Vec of `Project` structs backed by `<config_dir>/projects.json`. All mutations go through `save()` which does an atomic write (UUID-named `.tmp` then rename).

`list_projects` iterates the in-memory vec, clones each entry, and sets `is_missing` by checking whether the path still exists on the filesystem. This field is never persisted (`#[serde(skip)]`). The async version checks `tokio::fs::try_exists` and treats an I/O error as `false` (path unreachable = treat as missing).

`load()` is deliberately synchronous because it is called from `AppState::new()` during Tauri `.setup()`, before the async runtime is fully available to user code.

## Decisions made within Task scope

- Used `tokio::fs::try_exists` (returns `Result<bool>`) rather than `tokio::fs::metadata` followed by an existence check, because `try_exists` is the idiomatic async equivalent of `Path::exists()` and handles the same error semantics (non-NotFound errors surface as `Err`, which we collapse to `false` via `unwrap_or`).
- Kept the doc comment on `load()` additive (prepended new paragraphs) rather than replacing the existing "If the file does not exist..." prose, to preserve existing documentation.
