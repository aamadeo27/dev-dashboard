# T4.6 — RetentionPruner

## What was done

Implemented `runs/retention.rs` with a public `run` function that walks each project's `.claude/runs/` directory, collects terminal-status runs (Completed/Failed/Stopped), applies age pruning (delete runs older than `retention_days`) then size pruning (delete oldest-first while total size exceeds `retention_size_mb`), and a `start` function that drives a 24-hour background loop. Wired a fire-and-forget startup sweep and the timer into `lib.rs` after the OrphanReaper block. Added 10 unit tests covering all specified scenarios.

## Decisions

- Age pruning runs before size pruning so stale data is removed first, reducing the size pressure.
- Unreadable file sizes contribute 0 (conservative: avoid deleting a run because we failed to measure it).
- Startup sweep is fire-and-forget (not blocked); pruning is not startup-critical unlike orphan reaping.
- `start` uses `tokio::spawn` internally; the outer `tauri::async_runtime::spawn` just drives the async fn to completion.

## Test scenarios

- IT-RP-01 — integration — 31-day-old Completed run deleted when retention_days=30
- IT-RP-02 — integration — 5-day-old Completed run survives when retention_days=30
- IT-RP-03 — integration — Pending and Running runs never deleted (100 days old, tightest caps)
- IT-RP-04 — integration — 600 MB / 500 MB cap: oldest run (200 MB) deleted, remaining 400 MB survive
- IT-RP-05 — integration — Completed, Failed, Stopped all eligible for age pruning
- IT-RP-06 — integration — Missing runs dir does not panic
- IT-RP-07 — integration — Two separate projects each pruned independently in one sweep
- IT-RP-08 — integration — Size pruning oldest-first ordering: run-a (3d) deleted before run-b (2d) and run-c (1d)
