## Scope findings

[LOW] F-1: `read_attached_md` is declared `pub async fn` and is exposed via `pub mod commands` in `ipc/mod.rs`. The integration test legitimately uses it across the crate boundary, so the visibility is needed — but it widens the public API surface beyond what the task strictly requires. A `pub(crate)` visibility with a re-export only for tests (using `#[cfg(test)]`) would keep the surface smaller. Not blocking, but worth noting.

[LOW] F-2: `chrono` and `serde_json` are added to `[dev-dependencies]` in `Cargo.toml`, but both crates are already present in `[dependencies]`. In Cargo, dev-dependencies supplement production dependencies for test/bench targets, so duplicating entries that already exist in `[dependencies]` is redundant noise. The entries compile without error (Cargo deduplicates), but they should be removed.

[LOW] F-3: The integration test (IT-T4.4-09/-10) tests `Run` JSON serialization rather than calling `TranscriptWriter::create` directly (which is `pub(crate)` and inaccessible from the integration crate). The test comment acknowledges this explicitly. This is an acceptable scope compromise given the Tauri test-context limitation, and the serialization contract is the correct observable proxy. No action required — noted for completeness.

## Quality findings

[MED] F-4: Empty-content stdin write is not guarded. When `attached_md_content` is `Some(vec![])` (an empty file — accepted by `read_attached_md` per IT-T4.4-05), the code at step 7b still writes `b"\n"` to stdin. The `write_all(content)` on an empty slice is a no-op, but the trailing newline write proceeds unconditionally. The integration test comment for IT-T4.4-05 even states "The caller (`launch_run` step 7b) skips the stdin write when content is empty" — which is factually wrong; the code does not skip it. A lone `\n` on stdin as the first write before any real prompt may confuse the Claude CLI. A guard `if !content.is_empty()` should be added.

[LOW] F-5: TOCTOU window between `tokio::fs::metadata` and `tokio::fs::read` in `read_attached_md`. A file that is exactly at the limit at `metadata()` time could grow between the two calls and be read in full. The task doc itself notes this regression risk. In practice this is negligible, but the code could alternatively use `tokio::fs::read` first and then check the returned `Vec::len()`, eliminating the race entirely with no extra syscall overhead (a single allocation). Low severity — not blocking.

[LOW] F-6: IT-T4.4-02 ("no side effects") asserts that the temp dir is empty after the call, but `read_attached_md` never creates files — that invariant is entirely a `launch_run` contract, not a `read_attached_md` contract. The test verifies the correct behaviour but is testing the wrong unit for that assertion. If this test is ever run after a future refactor that adds a side-effect elsewhere in the same temp dir, the assertion will give a misleading signal. The assertion is not harmful today and does not need to change, but the comment could be more precise.

[LOW] F-7: IT-T4.4-07 message check uses `msg.chars().any(|c| c.is_ascii_digit())` as a fallback, which will pass as long as any digit appears in the message — including path components that contain digits (e.g., `/tmp/tmp1234/oversized.md`). The assertion effectively always passes on typical temp paths regardless of whether the error message actually mentions size. The test does not fail incorrectly, but the fallback weakens the assertion. Checking for `"too large"` or `"bytes"` without the digit fallback would be stricter.
