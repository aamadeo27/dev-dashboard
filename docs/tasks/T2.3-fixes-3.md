# T2.3-fixes-3: UTF-8 Safe String Truncation in git.rs + per-id Length Guard

## What was done

Two security fixes applied to the git poller subsystem:

**FIX-1 (MEDIUM):** Replaced byte-index string slicing (`s[..N]`) in `git_status_for_path` with a new `sanitize_git_string` helper that uses `.chars().filter(...).take(N).collect()`. Byte-index slicing panics at runtime when the slice boundary falls inside a multi-byte UTF-8 sequence (e.g. Cyrillic branch names like `feature/тест`).

**FIX-2 (LOW):** Added a per-id length check in `set_visible_projects` that rejects any id longer than 128 bytes before the ids are written to the visible-set mutex.

## How the affected components work

### `sanitize_git_string` (git.rs)

Pure helper with no side effects:

```rust
fn sanitize_git_string(s: &str, max_chars: usize) -> String {
    s.chars().filter(|c| !c.is_control()).take(max_chars).collect()
}
```

- Operates on Unicode scalar values (chars), not bytes — always lands on a valid char boundary.
- Used in two sites inside `git_status_for_path`:
  1. Error message from `git2::Repository::discover` failure (capped at 512 chars).
  2. Branch short name from `repo.head()` (capped at 256 chars).

### `set_visible_projects` per-id guard (commands.rs)

Added immediately after the existing `ids.len() > 1024` count guard:

```rust
if ids.iter().any(|id| id.len() > 128) {
    return Err(AppError::InvalidInput(
        "set_visible_projects: id too long (max 128 chars)".to_string()
    ));
}
```

`id.len()` is byte length. Project IDs are UUID v7 strings (36 bytes) so legitimate IDs are well under 128. The guard prevents oversized strings from accumulating in the visible-set HashSet.

## Decisions made within Task scope

- `id.len() > 128` uses byte length (not char count) because project IDs are ASCII-only UUID strings; byte and char counts are identical for them. Consistent with how the existing `ids.len() > 1024` count guard works.
- `sanitize_git_string` placed immediately before `git_status_for_path` (its only caller is in the same file).
