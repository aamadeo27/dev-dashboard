## How the affected components work

### AppError serialization

`AppError` implements `serde::Serialize` manually (not derived) so the wire shape is fixed regardless of variant internals. Each variant maps to a `code` string via `AppError::code()`:

```
AppError::NotFound(_)      -> "NOT_FOUND"
AppError::AlreadyExists(_) -> "ALREADY_EXISTS"
AppError::Io(_)            -> "IO"
AppError::Git(_)           -> "GIT"
AppError::Cli(_)           -> "CLI"
AppError::ParseError(_)    -> "PARSE_ERROR"
AppError::InvalidInput(_)  -> "INVALID_INPUT"
AppError::PermissionDenied(_) -> "PERMISSION_DENIED"
AppError::Internal(_)      -> "INTERNAL"
```

Wire shape: `{ "code": "NOT_FOUND", "message": "Not found: ...", "details": null }`. The `details` field is always `null` in v1; the field is included so the frontend type definition never needs a breaking change when details are added.

The `From<std::io::Error>` impl is derived via `thiserror`'s `#[from]` attribute on the `Io` variant. The `From<git2::Error>` impl is manual (in the file) because `git2::Error` is a foreign type.

### AppState

Holds two fields:
- `config_dir` ‚Äî the resolved `PathBuf` pointing to the app's config directory. All domain components (settings, registry, pruner) derive their sub-paths from this.
- `log_guard` ‚Äî the `tracing_appender::non_blocking::WorkerGuard` that must live as long as the process so the background log-writer thread is not dropped.

Stored in Tauri via `.manage(state)` in `lib.rs`, accessed in command handlers as `tauri::State<'_, AppState>`.

### Config dir resolution (lib.rs)

```
if DEV_DASHBOARD_CONFIG_DIR env var set:
    use that path directly
else:
    dirs::config_dir() + "dev-dashboard"
    (fallback: std::env::temp_dir() + "dev-dashboard" if dirs returns None)
```

The resolved path is logged at `tracing::info!` level before the Tauri builder runs. Logging is initialized against `<config_dir>/logs` before the `info!` call, so this line always lands in the file.

### ping command

A `#[tauri::command]` returning `Result<String, String>` (not `AppResult`) ‚Äî intentionally simple. Its only purpose is to verify the IPC channel is wired up end-to-end. The frontend can call `invoke("ping")` and expect `"pong"`.

## Decisions made within Task scope

- `AppError::code()` returns `&'static str` (not `String`) ‚Äî avoids an allocation per error at the only call site (the `Serialize` impl).
- Unit tests cover all 9 variant codes via a single table-driven test (`all_variant_codes_are_correct`) plus dedicated tests for `NotFound`, `Internal`, `Io` (the three acceptance criteria cases), and a shape test asserting exactly 3 fields.
- `From<git2::Error>` uses `e.message().to_owned()` ‚Äî `git2::Error::message()` strips trailing whitespace and returns the most user-readable description.
- No `unwrap()` in `run()` on the `dirs::config_dir()` call ‚Äî uses `unwrap_or_else(std::env::temp_dir)` per the Task spec.

All tests are under `#[cfg(test)] mod tests`. Level: unit (owned by Coder), reviewed and supplemented by Tester.

| # | Name | Preconditions | Steps | Expected result |

| U-4 | `all_variant_codes_are_correct` | ‚Äî | Serialize all 8 string-constructed variants | Each serialized `code` matches its SCREAMING_SNAKE_CASE constant (NotFound, AlreadyExists, Git, Cli, ParseError, InvalidInput, PermissionDenied, Internal) |

**Gap addressed by U-6**: The coder's 5 tests left `From<git2::Error>` untested. U-6 exercises the conversion path end-to-end via `git2::Error::from_str`, which is the lightest constructor that does not require a live repository.

### Integration tests (`src-tauri/tests/config_dir.rs`)

Level: medium/integration. These tests exercise the config directory resolution algorithm (reproduced verbatim from `lib.rs::run()`) against real environment variables. No external processes, no DB, no Tauri runtime ‚Äî just `std::env` and `dirs`.

**Isolation strategy**: a process-wide `Mutex<()>` (`ENV_LOCK`) serialises all env-var mutations across tests, so the default multi-threaded test runner is safe. Each test saves and restores the env var around its assertions.

| # | Name | Level | Preconditions | Steps | Expected result |
|---|---|---|---|---|---|
| I-1 | `config_dir_uses_env_var_override` | medium | `DEV_DASHBOARD_CONFIG_DIR` is unset (or saved/restored) | Set env var to an arbitrary temp path; call resolver | Returned path equals env var value exactly |
| I-2 | `config_dir_env_var_is_not_modified` | medium | Same | Set env var to a custom path without "dev-dashboard" suffix; call resolver | Returned path equals the raw env var ‚Äî no extra segment appended |
| I-3 | `config_dir_default_ends_with_dev_dashboard` | medium | `DEV_DASHBOARD_CONFIG_DIR` is unset | Remove env var; call resolver | Last path segment is `"dev-dashboard"` |
| I-4 | `config_dir_default_is_absolute` | medium | Same | Remove env var; call resolver | Returned path is absolute |
| I-5 | `config_dir_resolution_is_deterministic` | medium | Same | Remove env var; call resolver twice consecutively | Both calls return identical paths |
| I-6 | `config_dir_env_var_absolute_path_preserved` | medium | `DEV_DASHBOARD_CONFIG_DIR` is unset | Set env var to `temp_dir().join("dd-abs-test")`; call resolver | Result is absolute and equals the set value |
| I-7 | `config_dir_falls_back_to_temp_dir_when_dirs_returns_none` | medium | ‚Äî | **(IGNORED)** Cannot force `dirs::config_dir()` to return `None` without injecting the base dir | Requires refactor: extract `resolve_config_dir_with(base: Option<PathBuf>)` from `lib.rs::run()`; then call with `None` and assert result equals `temp_dir().join("dev-dashboard")` |

**Refactor needed for I-7**: extract a `pub(crate) fn resolve_config_dir_with(base: Option<PathBuf>) -> PathBuf` helper from `lib.rs::run()`. This separates the pure resolution logic from the side-effecting `dirs::config_dir()` call and makes the temp-dir fallback branch fully testable. The test body is stubbed with `todo!()` and the `#[ignore]` attribute; it will pass as soon as the refactor lands.
