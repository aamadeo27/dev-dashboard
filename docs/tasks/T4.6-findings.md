## Quality Findings

### High — blocks merge

**F-1 | HIGH | code-quality**
**Description**: `compute_total_size` is declared `async fn` but contains no `.await` and performs no I/O; the unnecessary `async` forces an extra future allocation on every size pass.
**Location**: `src-tauri/src/runs/retention.rs` — the `compute_total_size` function (diff line ~373)
**Fix**: Change signature to `fn compute_total_size(candidates: &[RunCandidate]) -> u64` and drop the `.await` at the one call site inside `prune_project`.

---

**F-2 | HIGH | code-quality**
**Description**: Size pruning counter advances even when `delete_run_dir` returns `false` (deletion failed). When a deletion fails the `idx += 1` still runs, so `total_size` is reduced by `dir_size` on a successful delete but the loop always advances past a failed entry. This is actually correct for advancing past the failed run, but the `total_size` subtraction does NOT happen when deletion fails — however `idx` still increments unconditionally, meaning the loop will skip past a failed entry and keep going. The real bug is more subtle: if a deletion fails, `total_size` is NOT decremented, so the loop correctly continues trying to prune the next candidate. This is acceptable. Re-reading the code:
```
if delete_run_dir(...).await {
    pruned += 1;
    total_size = total_size.saturating_sub(dir_size);
}
idx += 1;
```
The `idx` increment is outside the `if`, so on failure the entry is skipped but `total_size` is not reduced — the loop will continue to the next candidate, which is correct behaviour. This is actually fine. Withdrawing F-2 — no bug here.

*F-2 withdrawn after re-analysis.*

---

**F-3 | HIGH | code-quality**
**Description**: `measure_dir_size` only sums direct children that `is_file()`. If a run directory contains sub-directories (e.g. a nested artifact folder written by the CLI), those subtrees are silently excluded, causing the measured size to be lower than the actual size. The pruner therefore under-counts size and may fail to prune when the real on-disk footprint exceeds the cap.
**Location**: `src-tauri/src/runs/retention.rs` — `measure_dir_size` function (diff lines ~347-370)
**Fix**: Either use a recursive walk (e.g. `walkdir` crate or a recursive async helper) or document explicitly that run directories are guaranteed to contain only flat files. If the run layout is guaranteed flat by the `Run` contract, add a comment citing that guarantee; if it is not guaranteed, use a recursive walk.

### Medium — should fix

**F-4 | MED | code-quality**
**Description**: The helper `write_meta` is duplicated verbatim between the inline unit tests in `retention.rs` and the integration test file `retention_integration.rs`. The two copies will diverge when the `Run` struct changes.
**Location**: `src-tauri/src/runs/retention.rs` (inline tests, diff line ~418) and `src-tauri/tests/retention_integration.rs` (diff line ~696)
**Fix**: Move the helper to a shared `tests/common/mod.rs` (following the orphan integration test precedent) and re-export from both test crates. The inline unit tests can keep their own local copy only if the `Run` struct fields needed differ.

**F-5 | MED | code-quality**
**Description**: `start()` spawns a `tokio::spawn` task but the `JoinHandle` is immediately discarded (the call is a statement, not `let _ = tokio::spawn(...)`). If the runtime shuts down, there is no mechanism to observe or cancel the background loop. The outer `tauri::async_runtime::spawn` wrapper in `lib.rs` is also fire-and-forget. While this is acceptable per the task decision log, the inner `tokio::spawn` handle should at minimum be explicitly dropped with `let _ =` or stored, to make the intent visible and avoid a clippy `must_use` warning on `JoinHandle`.
**Location**: `src-tauri/src/runs/retention.rs` — `start()` function (diff line ~156)
**Fix**: Change `tokio::spawn(async move { ... });` to `let _ = tokio::spawn(async move { ... });` to silence the implicit must-use and clarify intent.

**F-6 | MED | code-quality**
**Description**: The 24h timer loop sleeps *before* the first tick, meaning the first periodic sweep does not fire until 24 hours after startup. The startup sweep handles the initial run, but if `start()` is called and the startup sweep completes before the first timer tick, there is a 24-hour window before the timer fires. This is architecturally acceptable (startup sweep covers it) but the `start()` doc comment does not explain this sleep-first behaviour, which will confuse future readers.
**Location**: `src-tauri/src/runs/retention.rs` — `start()` function (diff line ~158)
**Fix**: Add a comment above the `tokio::time::sleep` line explaining "sleep first: startup sweep already ran; first periodic sweep fires 24h later."

### Low / Nit

**F-7 | LOW | code-quality**
**Description**: `collect_terminal_runs` returns `None` (logged at warn) when `read_dir` fails for any reason other than `NotFound`, but returns `Some(vec![])` when the directory does not exist. The distinction is correct, but `prune_project` treats both `None` and `Some(vec![])` as "nothing to prune" — it cannot distinguish "directory unreadable" from "directory empty". This means an IO error on a project's runs dir silently skips that project with only a warn log and no increment to any error counter. Acceptable per design but warrants a comment.
**Location**: `src-tauri/src/runs/retention.rs` — `collect_terminal_runs` (diff line ~249) and `prune_project` (diff line ~188)
**Fix**: Add a comment in `prune_project` at the `None => return 0` arm: "// runs dir unreadable — warn already emitted in collect_terminal_runs; skip project."

**F-8 | NIT | code-quality**
**Description**: `projects_processed` is computed as `project_paths.len()` before the loop but the variable name implies "projects successfully processed". It counts input projects, not projects that were not skipped due to errors.
**Location**: `src-tauri/src/runs/retention.rs` — `run()` function (diff line ~139)
**Fix**: Rename to `projects_total` or `project_count` (the tracing field at the start already uses `project_count`) and use the same name for the finish log field for consistency.
