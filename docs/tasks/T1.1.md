# T1.1 — SettingsStore (backend)

## What was implemented

- `SettingsStore` struct with `load`, `save`, and `patch` methods in `src-tauri/src/settings/mod.rs`.
- `Default` impl on `Settings` with canonical defaults (git_poll=10, usage_poll=60, retention_days=30, retention_size_mb=500, view_mode=Grid).
- Corrupt-settings recovery: log warning via `tracing::warn!`, rename `settings.json` to `settings.json.broken`, return defaults.
- Atomic write: serialize to `settings.json.tmp` in same directory, then `std::fs::rename` over the target.
- Range validation in `patch()` before any mutation: returns `AppError::InvalidInput` on out-of-range values.
- `Arc<tokio::sync::Mutex<SettingsStore>>` field added to `AppState`.
- `SettingsStore::load()` called during `run()` startup in `lib.rs` before `manage(state)`.
- Two new Tauri commands: `get_settings` and `update_settings`, registered in `invoke_handler`.
- TypeScript wrappers `getSettings()` and `updateSettings()` in `src/ipc/commands.ts`, typed against `bindings.ts`.
- `tempfile = "3"` dev-dependency added to `Cargo.toml` for tests.
- 8 unit tests covering all acceptance criteria.

## How the affected component works

### Data flow

```
Frontend invoke("get_settings")
  → commands::get_settings()
      → AppState.settings.lock().await
      → SettingsStore.settings().clone()
      → serialized to JSON over IPC

Frontend invoke("update_settings", { patch })
  → commands::update_settings(patch, state)
      → AppState.settings.lock().await (exclusive)
      → SettingsStore.patch(patch, config_dir).await
          → validate path fields (absolute, exists, is_file/is_dir)
          → validate numeric ranges (AppError::InvalidInput on failure)
          → apply non-None fields
          → SettingsStore.save(config_dir).await
              → serde_json::to_string_pretty
              → tokio::fs::write → settings.json.tmp
              → tokio::fs::rename → settings.json
      → return updated Settings
```

### Startup path

`lib.rs::run()` resolves `config_dir`, initializes logging, then calls `SettingsStore::load(&config_dir)`. On first launch (no settings.json present), defaults are returned and immediately written to disk using `std::fs::write` (sync, before the Tokio runtime starts). This guarantees `settings.json` always exists after the first startup.

### Key contracts

- `patch()` validates ALL fields before mutating any — no partial-mutation on error.
- `save()` is async (uses `tokio::fs`) and atomic: a crash between write and rename leaves `settings.json.tmp` behind but the original `settings.json` is untouched.
- The Tokio mutex ensures at most one concurrent writer; readers (`get_settings`) also take the lock to avoid torn reads.
- `SettingsStore.settings` field is private; accessed via the `settings()` getter returning `&Settings`.
- Path fields (`claude_cli_path`, `parent_dir`) are validated for absolute paths, existence, and type (file/dir) before any mutation or save.
- `retention_days` is bounded [1, 90]; `retention_size_mb` is bounded [50, 10_240].
- Path existence/type checks in `patch()` use `tokio::fs::metadata` (async, non-blocking) — no `std::fs` predicates inside async context.

## Decisions made within Task scope

- First-launch write uses `std::fs::write` directly in `lib.rs::run()` (sync context, before the Tokio runtime). `save()` is async, so cannot be called here; the sync write is appropriate and avoids double-buffering.
- `SettingsPatch::parent_dir` and `claude_cli_path` use `if let Some(v)` to apply, meaning `None` in the patch leaves the current value intact. There is no mechanism to explicitly clear them to `None` via a patch — the task spec did not require it.
- `PartialEq` added to `ViewMode` (not present in original stub) to enable `assert_eq!` in tests. This is a non-breaking additive derive.
- `SettingsPatch` derives `Default` (all-None via `#[derive(Default)]`), replacing both the manual `impl Default` block and ad-hoc `empty_patch()` helpers.
- `SettingsStore.settings` field is private with a `settings() -> &Settings` getter to prevent callers from mutating settings outside of `patch()`.
- Unit tests that call `patch` or `save` converted from `#[test]` to `#[tokio::test]`; the one test that only calls `load` (sync) and reads filesystem state is kept as `#[test]`.

## Acceptance criteria status

- First launch creates a default settings file — `settings.json` written at startup via `std::fs::write` if absent.
- `get_settings` command works — reads locked store, returns clone of Settings via `settings()` getter.
- `update_settings` command works — validates path/numeric fields, patches, saves (async), returns updated Settings.
- Corrupted settings file falls back to defaults — `tracing::warn!` logged, file renamed to `settings.json.broken`.
- Path fields validated before persistence: must be absolute, `claude_cli_path` must exist and be a file (async `tokio::fs::metadata`), `parent_dir` must be a directory if it exists (async `tokio::fs::metadata`).
- Upper bounds enforced: `retention_days` max 90, `retention_size_mb` max 10_240.
