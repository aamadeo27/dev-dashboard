# T1.6 — cli:lost Detection Background Task

## What was done

Added a Rust background task (`cli_watcher`) that continuously monitors whether the Claude CLI binary is still reachable after app startup. When the binary transitions from found to not-found, the task emits a `cli:lost` Tauri event (consumable by the frontend) and logs a structured warning matching monitoring.md §2.13j.

## How the affected component works

### Data flow

```
lib.rs .setup() hook
  └─ ipc::cli_watcher::start(app_handle, settings)
        ├─ registers on_window_event listener → writes AtomicBool `is_focused`
        └─ tokio::spawn(async loop)
              ├─ waits 60 focused-window seconds (5s tick, resets while unfocused)
              ├─ reads settings.claude_cli_path via Mutex<SettingsStore>
              ├─ calls resolve_cli_path(None, settings_path) → PathBuf
              ├─ calls probe_cli(path) → bool (5s timeout, discards output)
              └─ on found→not-found transition:
                    tracing::warn!(component="cli_detect", last_known_path, "cli lost mid-session")
                    app_handle.emit("cli:lost", path_str)
```

### Key functions

**`start(app_handle, settings)`** — public entry point. Registers the focus listener and spawns the async poll loop. Returns immediately; the loop runs until the process exits.

**`probe_cli(path) -> bool`** — async. Spawns `<path> --version` with all stdio discarded. Returns `true` if the process can be spawned (and either exits within 5s or times out — timeout is treated as "binary exists"). Returns `false` on spawn error (file not found, permission denied, etc.).

### State machine

| `last_found` | `found` (this tick) | Action |
|---|---|---|
| `true` | `true` | Update `last_known_path`. No event. |
| `true` | `false` | **Transition**: log warn, emit `cli:lost`, set `last_found=false`. |
| `false` | `true` | **Restoration**: log info, set `last_found=true`. |
| `false` | `false` | No action (already notified). |

### Focus-pause behaviour

The poll loop uses a 5-second sub-tick. Each tick that finds the window focused increments a counter; when unfocused the counter resets to 0. This means 60 seconds of continuous focused time must elapse between probes — the window going in and out of focus restarts the 60s countdown.

### Contracts

- `last_found` starts as `true`: assumes the CLI was present at launch (T1.3 startup check). If CLI was absent at launch and the app reached Dashboard anyway, the first poll cycle will emit `cli:lost` correctly.
- `cli:lost` is emitted **at most once** per found→not-found transition. Subsequent polls while CLI is still absent are silent.
- Probe timeout: 5 seconds (half the interactive 10s used by `verify_claude_cli`).
- `resolve_cli_path(None, settings_path)` is called fresh each poll cycle, so changes to `claude_cli_path` in settings are picked up automatically.

## Decisions made within Task scope

- **Focus counter resets on unfocus** (not pauses): Task spec says "resets the countdown", meaning unfocus is not neutral — it actively restarts the 60s. This matches the spec phrase "on refocus, the 60s starts fresh."
- **Probe timeout treated as found**: A 5-second timeout means the binary launched (spawn succeeded) but is slow to respond. Treating a slow-but-launchable binary as "found" prevents spurious `cli:lost` events when the CLI is under load. This matches the intent of "checks if the binary is runnable."
- **`last_known_path` updated only on found**: The variable holds the last path that successfully probed, not every resolved path. This means the warning message reports the path that was last known to work, which is more useful for debugging.
- **Alphabetical module ordering** in `ipc/mod.rs`: `cli_watcher` before `commands` before `events`, consistent with existing convention.

## How to test / verify

### Unit tests (no Tauri runtime needed)

```
cargo test -p dev-dashboard-lib ipc::cli_watcher
```

Tests included:
- `probe_cli_returns_false_for_nonexistent_path` — spawn failure → false
- `probe_cli_returns_false_for_directory` — directory not executable → false
- `focus_accumulator_resets_when_unfocused` — counter logic simulation
- `watcher_path_resolution_uses_settings_path` — delegates to resolve_cli_path
- `watcher_path_resolution_falls_back_to_claude` — no settings path fallback
- `path_str_uses_resolved_when_last_known_path_is_none` — warning path fallback
- `path_str_prefers_last_known_path_when_set` — warning prefers cached path

### Manual smoke test

1. Build and run the app: `cargo tauri dev`
2. Verify the CLI is detected at launch (no `cli:lost` in logs).
3. Rename/move the Claude CLI binary.
4. Wait up to 60 seconds with the window focused.
5. Confirm in the log: `WARN cli_detect: cli lost mid-session last_known_path=...`
6. Confirm the frontend receives `cli:lost` event (check browser devtools if applicable).
7. Restore the CLI binary.
8. Wait up to 60 more seconds.
9. Confirm in the log: `INFO cli_detect: cli restored mid-session`.

### Log format verification

The warning log must match:
```
WARN dev_dashboard: cli lost mid-session component="cli_detect" last_known_path="/path/to/claude"
```

### Integration test file

`src-tauri/tests/cli_watcher_integration.rs` is **not created**. `probe_cli` is a private `async fn` — it cannot be imported from outside the crate. All meaningful testable behavior is covered by the inline `#[cfg(test)]` tests above. The public surface of `cli_watcher` is only `start()`, which requires a Tauri `AppHandle` and cannot be constructed without a running Tauri application.
