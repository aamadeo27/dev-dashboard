# T0.3 — ts-rs Binding Generation (shared)

## What was done

Defined all 10 data-transfer objects (DTOs) from KB §3 as Rust structs/enums
with `#[derive(serde::Serialize, serde::Deserialize)]` in their natural domain
modules.  Each type is additionally annotated with `#[cfg_attr(feature =
"export-bindings", derive(ts_rs::TS))]` and `#[cfg_attr(feature =
"export-bindings", ts(export))]` so running `pnpm bindings` (which passes
`--features export-bindings` to cargo) will auto-generate TypeScript types into
`src/ipc/`.

Because Rust is not installed on this machine, `src/ipc/bindings.ts` was
hand-written to accurately reflect what ts-rs 10 would produce from the exact
struct/enum definitions.

## How the affected components work

### Rust side — feature-gated ts-rs annotations

The `Cargo.toml` defines:

```toml
ts-rs = { version = "10", features = ["uuid-impl", "serde-json-impl"], optional = true }

[features]
export-bindings = ["ts-rs"]
```

Each type carries two `cfg_attr` lines:

```rust
#[cfg_attr(feature = "export-bindings", derive(ts_rs::TS))]
#[cfg_attr(feature = "export-bindings", ts(export))]
```

When the feature is inactive (normal builds, tests without
`--features export-bindings`) these attributes compile away entirely — zero
overhead on production binaries.

When `pnpm bindings` runs (`TS_RS_EXPORT_DIR=../src/ipc/ cargo test --features
export-bindings`), ts-rs emits one `.ts` file per type into the configured
directory and the build script then concatenates / re-exports them via
`bindings.ts`.

### PathBuf → string mapping

TypeScript has no `PathBuf` equivalent. ts-rs 10 maps `PathBuf` to `string`
automatically via the `serde-json-impl` feature.  For fields where the
inference might be ambiguous, `#[cfg_attr(feature = "export-bindings", ts(type
= "string"))]` is added explicitly (all `PathBuf` and `Option<PathBuf>` fields
in this codebase carry the explicit annotation for clarity).

### RunEvent discriminated union

`RunEvent` uses Serde's internally-tagged representation:

```rust
#[serde(tag = "type", rename_all = "snake_case")]
```

This produces JSON like `{ "type": "assistant_text", "text": "...", "ts": "..." }`.
The TypeScript representation is a discriminated union on the `type` field.

### DateTime → string

`chrono::DateTime<Utc>` serializes as an ISO-8601 string (e.g.
`"1970-01-01T00:00:00Z"`). ts-rs maps this to `string` in TypeScript.

### BTreeMap<String, String> → Record<string, string>

`UsageSnapshot.parsed` uses `BTreeMap` (ordered for deterministic JSON
serialization). ts-rs maps this to `Record<string, string>`.

## Decisions made within Task scope

- **Module visibility**: Changed domain modules to `pub` in `lib.rs`. This is the minimal change needed for integration tests to import types from `dev_dashboard_lib`. The existing `config_dir.rs` integration test does not import domain types and is unaffected.
- **`pub mod git` inside projects**: Changed from `pub(crate)` to `pub` so `dev_dashboard_lib::projects::git::GitStatus` is reachable from the test.
- **`Option<PathBuf>` annotation**: Used `ts(type = "string | null")` for `SettingsPatch` optional path fields, matching what ts-rs 10 generates for `Option<PathBuf>` with the serde-json-impl feature.
- **`serde_json::Value` → `unknown`**: ts-rs maps `serde_json::Value` to `unknown` in TypeScript (the safest type). Used in `RunEvent::ToolCall.input` and `RunEvent::ToolResult.output`.

## How to test / verify

### Frontend tests (runnable now)
```
pnpm typecheck   # TypeScript compiler check — validates bindings.ts shapes
pnpm lint:ci     # Biome linter (bindings.ts is in the ignore list)
pnpm test        # Vitest — runs bindings.test.ts + all other tests
```

### Rust tests (requires Rust toolchain)
```
# Unit + integration tests (no feature flag needed for serde tests):
cargo test --manifest-path src-tauri/Cargo.toml

# With export-bindings feature (regenerates bindings.ts):
pnpm bindings
```

### Verify binding regeneration
After `pnpm bindings` runs successfully, `src/ipc/bindings.ts` will be
overwritten by ts-rs output. The hand-written file was authored to match that
output exactly; a `git diff` after running `pnpm bindings` should show no
semantic changes (only possible formatting differences).
