### Key contracts

- `SettingsPatch` has `string | null` for path fields and `number | null` for numeric fields. `null` means "no change" for numeric fields in the backend. Path fields: empty string in the form â†’ `null` in patch (via `emptyToNull`). The backend interprets `null` as "clear the path".
- `viewMode` in the draft is initialized from `useUiStore` (not from the backend response) so it always reflects the current in-memory state. The backend response is the source of truth for numeric/path fields only.
- `isDirty` compares: path fields via `emptyToNull`, numeric fields via `parseInt`, `viewMode` via direct comparison.
- `canSave` requires no validation errors and `isSaving === false`.

## Decisions made within Task scope

- **`viewMode` source of truth**: on mount, view mode is read from `useUiStore.getState().viewMode` (not a subscription, read once inside the init effect) rather than from the backend `Settings` object. This keeps the setting consistent with whatever the Dashboard toolbar last set.
- **`isDirty` comparison for `viewMode`**: `draft.viewMode !== settings.view_mode` is still checked so that if a user toggles and then reverts to the original backend value, the form is correctly marked clean.
- **Unsaved-changes prompt**: uses `window.confirm` on the Back button click and `window.addEventListener("beforeunload", ...)` for OS-level close.
- **Inline style objects**: all styles use `React.CSSProperties` objects referencing CSS variables from `tokens.css`. No hardcoded color values anywhere.
- **`clampInt` in save handler**: even though validation blocks saving when a field is invalid, `clampInt` is applied when constructing the patch as a belt-and-suspenders measure.
- **`initializedRef` as `useRef`**: switching from `useState(false)` to `useRef(false)` eliminates the spurious re-render that occurred when `initialized â†’ true` fired simultaneously with `setDraft`. The ref is a one-time gate; it never needs to trigger a render.
- **CLI path fieldset split**: the old "Paths" fieldset was split into "Claude CLI" (with Browse + Verify) and "Projects" (parent directory only) to give the CLI section room for the verify affordances without making the fieldset visually overloaded.
