# T4.1 — EventParser

## What was done

Implemented `EventParser` in `src-tauri/src/runs/parser.rs` — a stateful, line-buffered heuristic parser that converts Claude CLI stdout byte chunks into structured `RunEvent` values. A companion `patterns.rs` submodule holds all sentinel strings and version constants.

## How the affected component works

### Data flow

```
bytes (arbitrary chunks)
  → EventParser::feed()
      → line_buf accumulation until '\n'
          → process_line()
              → state dispatch (Idle / AccumulatingToolCall / AccumulatingThinking /
                                AccumulatingToolResult / AccumulatingDiff)
                  → RunEvent variants
```

### State machine

`ParserState` is an internal enum with five variants:

| Variant | Opened by | Closed by | Emits |
|---|---|---|---|
| `Idle` | initial state, after any block ends | — | one event per line |
| `AccumulatingToolCall { name, input_lines }` | `⏺ Tool: <name>` line | empty line or non-indented line | `ToolCall` (or `FileEdit` if edit tool + diff) |
| `AccumulatingThinking { lines, needs_close_tag }` | `<thinking>` or `✻ Thinking` | `</thinking>` (tag form) or empty/non-indented (alt form) | `Thinking` |
| `AccumulatingToolResult { lines }` | `⎿` prefix or `Result:` keyword | empty line or non-indented line | `ToolResult` |
| `AccumulatingDiff { tool_name, diff_lines }` | `⏺ Tool: Edit/Write` + `@@` hunk header | empty or non-diff line | `FileEdit` |

`Idle` is handled with an early-return pattern before the main `match &mut self.state` to avoid a double-mutable-borrow on `self` (the Idle arm needs `&mut self` for `process_idle_line`; non-Idle arms use static helpers that borrow only the variant fields).

### Allocation strategy

- `line_buf: Vec<u8>` is allocated once at construction (`with_capacity(256)`) and reused across all `feed()` calls via `clear()`.
- No per-byte heap allocations in the hot path; the only allocations are `String::to_owned()` once per complete line, and `Vec<RunEvent>` growth.
- Block state `Vec<String>` fields grow during accumulation and are consumed (not leaked) when the block closes.

### Error handling

- Invalid UTF-8 in a line → `RunEvent::System` with lossy-decoded text; never panics.
- Malformed JSON in a tool-call body → `RunEvent::System` with the raw body; state returns to `Idle`.
- All other unrecognised lines → `RunEvent::AssistantText`.

### StepFailed guard

`STEP_FAILED_SENTINEL` is `""`. The guard `!sentinel.is_empty()` prevents any line (including the empty string) from triggering `RunEvent::StepFailed`. T4.8 will set the real sentinel value.

## Decisions made within task scope

- **`new_no_log()` private constructor**: used internally when a block-end line needs to be re-dispatched through Idle logic without a `self` borrow conflict and without emitting a spurious `tracing::info!`. This is a detail invisible to callers.
- **`process_idle_line` as `&mut self` method**: consistent receiver type; does not actually mutate `self` but is called in a context that requires a mutable reference.
- **Static handler methods** (`process_tool_call_line`, etc.) take individual `&mut` field references rather than `&mut self` to satisfy the borrow checker when called from inside `match &mut self.state`.
- **`splitn(2, ...)` for XML tag parsing**: avoids regex; splits on the tag literal once and discards the rest.
- **`trim_start()` on diff body lines**: indented input lines like `"  +added"` are stripped before the `+`/`-` prefix check so `count_diff_lines` works correctly.

## Fix pass 1

Applied 2026-05-23 against security and quality findings in the fix-pass spec.

| Fix | What changed |
|-----|-------------|
| FIX-1 | Added `const MAX_LINE_BYTES: usize = 1_048_576` at module level. In `feed()`, the `else` branch that pushes to `line_buf` now checks `line_buf.len() >= MAX_LINE_BYTES` before pushing. If exceeded: emits `RunEvent::System` with text `"[parser] oversized line truncated at 1048576 bytes"`, clears `line_buf`, resets `self.state = ParserState::Idle`, and skips the current byte. |
| FIX-2 | Added `const MAX_BLOCK_LINES: usize = 10_000` and `const MAX_BLOCK_BYTES: usize = 4_194_304` at module level. In the push-to-accumulator path of `process_tool_call_line`, `process_thinking_line`, and `process_tool_result_line`, a check is added before `push`: if `lines.len() >= MAX_BLOCK_LINES || total_bytes + line.len() > MAX_BLOCK_BYTES` → emit `RunEvent::System { text: "[parser] block truncated: too many lines/bytes" }` and return `Some(ParserState::Idle)`. The `AccumulatingDiff` accumulator was not guarded because that state was removed in FIX-10. |
| FIX-3 | In `process_tool_call_line` (non-edit path), a size guard was added before `serde_json::from_str(&raw_input)`: if `raw_input.len() > MAX_BLOCK_BYTES` emit a System event and return Idle (with redirect if `!is_empty`). Same guard added in `process_tool_result_line` before `serde_json::from_str(&raw_output)`. |
| FIX-4 | Added `fn dispatch_idle_line(line: &str, events: &mut Vec<RunEvent>) -> Option<ParserState>` free function that constructs a no-log tmp parser and calls `process_idle_line`, returning the `Option<ParserState>`. Replaced all 3 active redirect sites (`process_tool_call_line` malformed-body path, `process_tool_call_line` normal block-end path, `process_tool_result_line` block-end path) with `dispatch_idle_line(line, events).or(Some(ParserState::Idle))`, propagating the returned state so that e.g. a tool-call header that closes a prior block correctly opens the new `AccumulatingToolCall` state on `self`. The `process_diff_line` site was removed together with the entire method in FIX-10. |
| FIX-5 | Added `fn extract_diff_path(input_lines: &[String]) -> String` free function that scans accumulator lines for `--- a/<path>` or `+++ b/<path>` headers and returns the path. In `process_tool_call_line`, `path: tool_name.clone()` replaced with `path: extract_diff_path(input_lines)`. `FileEdit.path` now contains the actual file path instead of the tool name. |
| FIX-6 | Both `uuid::Uuid::new_v4()` calls replaced with `uuid::Uuid::now_v7()`: the ToolCall id in `process_tool_call_line` and the ToolResult call_id in `process_tool_result_line`. The `uuid` crate's `v7` feature was already declared in `Cargo.toml`. |
| FIX-7 | Added `fn strip_control_chars(s: &str) -> String` free function that filters out C0 control chars (except tab/LF/CR) and C1 chars (0x80–0x9F). Applied to: `AssistantText.text` in `process_idle_line`, `Thinking.text` in both `process_thinking_line` close paths and inline-close in `process_idle_line`, `ToolResult.output` plain-text fallback (the `serde_json::Value::String(...)` path), and `System.text` in the invalid-UTF-8 path in `feed()` and in malformed-block System events. `ToolCall.input` (a `serde_json::Value`) and `FileEdit.diff` were intentionally excluded. |
| FIX-8 | Added `#[inline] fn is_indented(line: &str) -> bool` free function. Replaced all 4 inline `line.starts_with(' ') || line.starts_with('\t')` duplicates in `process_tool_call_line`, `process_thinking_line`, `process_tool_result_line`, and `process_diff_line` (the last one was removed in FIX-10). |
| FIX-9 | In `process_tool_call_line`, `is_edit && has_diff_hunk` branch: changed `let diff = raw_input.clone()` to `let diff = raw_input` (move). `raw_input` is not used elsewhere in that branch so no clone is needed. |
| FIX-10 | Removed `AccumulatingDiff { tool_name, diff_lines }` enum variant, `process_diff_line` static method, and the `AccumulatingDiff { .. } => Self::process_diff_line(...)` match arm in `process_line`. Diff detection and `FileEdit` emission happen inline inside `process_tool_call_line` — the `AccumulatingDiff` state was never actually transitioned into from the live codebase. |
| patterns.rs NOTE | Added `// TODO(KB§9-item1): validate all patterns below against live Claude CLI output / before T4.8 ships; no baseline has been established yet.` comment after `PATTERN_VERSION` constant. |
| process_idle_line NOTE | Added comment explaining that `RunEvent::UserInput` is not emitted here — it is injected by the T4.3 caller (RunSession) when sending input to process stdin. |

### Contracts updated by this pass

- `FileEdit.path` now contains the actual file path extracted from diff headers (`--- a/` or `+++ b/`), not the tool name (`"Edit"` / `"Write"`). Callers that previously received `"Edit"` as the path will now receive the real file path.
- Adjacent tool-call headers without a blank separator now correctly produce two `ToolCall` events instead of one (Beta was previously dropped). Callers relying on the drop behavior will need to be updated.
- UUID fields on `ToolCall.id` and `ToolResult.call_id` are now v7 (time-ordered) instead of v4 (random). Format is still a standard UUID string; no wire-format change.
