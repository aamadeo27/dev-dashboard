# T2.1-fixes — ProjectRegistry CRUD Review Fixes

## What was done

Applied 7 post-review fixes to the T2.1 ProjectRegistry implementation. The fixes address a persistence bug (is_missing written to JSON), async safety (blocking I/O inside a held tokio Mutex), a release-build security surface (rename_project exposed in release handler), monitoring field naming, a tmp-file symlink attack vector, a tautological TS test, and unnecessary `#[tokio::test]` on pure-logic tests.

## How the affected component works

`ProjectRegistry` is an in-memory store of `Project` entries backed by `<config_dir>/projects.json`. The store is held behind a `tokio::sync::Mutex<ProjectRegistry>` in `AppState`. All CRUD methods (`add_project`, `remove_project`, `relocate_project`, `set_project_tags`, `rename_project`) are now `async fn` and call `tokio::fs` operations exclusively, so no tokio worker thread is blocked while the mutex is held.

**Save flow**: `save()` serializes `self.projects` to JSON, writes to a UUID-named `.tmp` file (`projects.<uuid>.tmp`), then atomically renames it to `projects.json`. The UUID suffix prevents symlink pre-creation attacks.

**is_missing**: The field carries `#[serde(skip)]`, so serde neither serializes nor deserializes it. `list_projects()` computes `!path.exists()` live on every call. The field is always `false` on a freshly-returned `Project` from a mutation method.

**canonicalize_dir**: Now async. Uses `tokio::fs::canonicalize` (existence + symlink resolution) followed by `tokio::fs::metadata` (is_dir check). Returns `AppError::InvalidInput` for NotFound and not-a-directory cases.

**IPC callers**: All six Tauri command functions in `commands.rs` now `.await` the registry method they call. The mutex lock is acquired, the async method runs, and the lock is released on method return — no blocking calls occur inside the lock window.

**rename_project exposure**: The command is registered only in the `#[cfg(debug_assertions)]` handler block. The `#[cfg(not(debug_assertions))]` block no longer includes it, preventing end-users from invoking an internal-only command.

## Decisions made within Task scope

- **UUID over PID for tmp filename**: The review allowed either. UUID is more robust against multiple concurrent saves (possible in future). `uuid::Uuid::new_v4()` was already a dependency.
- **`#[serde(skip)]` over `#[serde(skip_serializing)]`**: Per review instructions — skip on both serialize and deserialize so a legacy file containing `"is_missing"` is tolerated gracefully.
- **`list_projects` remains sync**: The method only calls `path.exists()` which is a stat; it does not hold the mutex (it operates on `&self` with no lock). The review noted wrapping in `spawn_blocking` or making it async were both acceptable; keeping it sync avoids introducing a new async surface where none is needed.
- **Integration test TC-PR-INT-01 tmp assertion**: Changed from checking `projects.tmp` (fixed name, now gone) to checking that no file matching `projects.*.tmp` pattern remains after save.

## How to test / verify

**TypeScript tests** (should show 283 passing):
```sh
pnpm test --run
```

**Rust unit tests** (requires Rust toolchain):
```sh
cargo test -p dev-dashboard-lib
```

**Rust integration tests**:
```sh
cargo test -p dev-dashboard-lib --test project_registry_integration
```

**Manual verification of is_missing**:
1. Build and launch the app.
2. Add a project.
3. Check `<config_dir>/projects.json` — the key `"is_missing"` must be absent.
4. Delete the project directory from disk.
5. Call `list_projects` — the returned project must have `is_missing: true`.

**Manual verification of rename_project release exclusion**:
1. Build in release mode (`cargo tauri build`).
2. Attempt to invoke `rename_project` from devtools — must return "command not found" error.
3. Build in debug mode — the command must be callable.
