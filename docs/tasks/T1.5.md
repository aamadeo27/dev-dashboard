# T1.5 — open_logs_folder command (backend)

## What was implemented

- New Tauri command `open_logs_folder` in `src-tauri/src/ipc/commands.rs` that:
  1. Constructs the logs path as `AppState.config_dir.join("logs")`.
  2. Creates the directory with `tokio::fs::create_dir_all` (async, idempotent).
  3. Opens the directory in the OS file manager via `tauri_plugin_opener::open_path`.
  4. Maps any opener error to `AppError::Io` via `std::io::Error::new(Other, ...)`.
- Command registered in `invoke_handler!` in `src-tauri/src/lib.rs`.
- TypeScript wrapper `openLogsFolder()` in `src/ipc/commands.ts`.
- 5 unit tests for the TS wrapper added to `src/ipc/commands.test.ts`.

## Acceptance criteria status

- Works on Win/macOS/Linux: `tauri_plugin_opener::open_path` is cross-platform; already registered plugin handles each OS.
- Errors surface as `AppError::Io`: opener errors are mapped with `std::io::Error::new(ErrorKind::Other, e.to_string())`.
- Depends on T0.6 (tauri-plugin-opener already initialized in lib.rs at line 57): satisfied.

## How the affected component works

### Data flow

```
Frontend: openLogsFolder()
  → invoke("open_logs_folder")
  → commands::open_logs_folder(state)
      → logs_dir = state.config_dir.join("logs")
      → tokio::fs::create_dir_all(&logs_dir).await   // ensures dir exists
      → tauri_plugin_opener::open_path(logs_dir, None::<&str>)
          → OS file manager opens the folder
          → Ok(()) on success
          → Err(e) → AppError::Io(std::io::Error::new(Other, e.to_string()))
```

### Key design decisions (within task scope)

- `open_path` is used rather than `reveal_item_in_dir`. The task spec specifies `open_path`. Both are available in `tauri_plugin_opener`; `open_path` opens the directory itself (which on all three platforms opens it in the file manager when the path is a directory).
- `None::<&str>` for the `with` parameter: lets the OS pick its default file manager.
- `create_dir_all` runs before `open_path`: the logs directory may not yet exist on first launch. Creating it first avoids an error from the opener.
- The command does not take an `AppHandle` parameter. The confirmed `tauri_plugin_opener::open_path` signature is `pub fn open_path<P: AsRef<Path>, S: AsRef<str>>(path: P, with: Option<S>) -> Result<(), Error>` — no handle required.

## Test scenarios

### Coverage rationale

`open_logs_folder` is a thin command shim. Its three testable behaviors are:

1. **Path construction** — `config_dir.join("logs")` uses the literal string `"logs"`.
2. **Directory creation** — `tokio::fs::create_dir_all` runs before `open_path`.
3. **Error mapping** — opener errors become `AppError::Io` with the original message preserved.

The fourth behavior — that `tauri_plugin_opener::open_path` actually opens the OS file manager — cannot be unit-tested without a live Tauri runtime and is covered by manual smoke testing. There is no meaningful integration or E2E test to add beyond the scenarios below.
