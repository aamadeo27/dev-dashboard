# T1.2 — verify_claude_cli command (backend)

## What was done

Added the `verify_claude_cli` Tauri command and its supporting `resolve_cli_path` helper.
The command probes the Claude CLI binary at the resolved path by spawning `<path> --version`,
capturing stdout/stderr, and returning a `CliCheck` struct. Errors are soft — spawn failures
populate `CliCheck.error` rather than returning an `AppError`. Added the TypeScript wrapper
`verifyClaudeCli()` and registered the command in the invoke handler.

**T1.2-sec-fix** (applied after review): Fixed four security findings in `verify_claude_cli`:

1. **HIGH — Unvalidated path_override**: Added `validate_cli_path(path: &Path) -> Result<(), String>`
   (async, uses `tokio::fs::metadata`). Checks: absolute path, no UNC on Windows, file exists
   and is a regular file. Called only on `path_override`-sourced paths; settings-sourced and
   `"claude"` fallback are exempt (see comments in code for rationale). Validation failure
   returns a soft `CliCheck { found: false, error: Some(...) }`.

2. **MEDIUM — No timeout**: Wrapped `child.wait_with_output()` in `tokio::time::timeout(10s)`.
   Timeout returns `CliCheck { found: false, error: "CLI probe timed out after 10 seconds" }`.
   Added `.stdin(Stdio::null())` to prevent the child from blocking on stdin reads.

3. **MEDIUM — Unbounded/unsanitized output**: Version string is now capped at 256 chars and
   stripped of control characters via `.filter(|c| !c.is_control()).take(256)`. Added the
   `time` feature to the tokio dependency in `Cargo.toml`.

4. **LOW — Log path with `%` format**: Changed `path = %resolved.display()` to `path = ?resolved`
   (Debug formatting) in both `tracing::info!` calls, so control characters in a crafted path
   are escaped rather than passed through verbatim.

**T1.2-monitoring-fixes** (applied after iteration 2 review): Fixed six review findings:

1. **G-1/G-2 — Monitoring log format (HIGH)**: Replaced the pre-spawn `tracing::info!` (which
   logged before the outcome was known) with outcome-specific log calls matching monitoring.md §2.13:
   - Success: `tracing::info!(component = "cli_detect", resolved_path = ?resolved, version = ..., mode = "interactive", "cli detected")` in the `Ok(Ok(output))` arm.
   - All failure paths: `tracing::warn!(component = "cli_detect", kind = "cli", path_tried = ?resolved, message = %error_msg, "cli detect failed")` in validate failure, spawn `Err`, timeout, and `wait_with_output` `Err` arms.

2. **SC-2 — Removed T1.5 tests from T1.2 test module**: Tests tagged T1.5
   (`logs_dir_for`, `opener_err_to_app_error`, `create_dir_all_*`, `opener_error_*`)
   removed from the `#[cfg(test)] mod tests` block. Their `#[cfg(test)]` helper functions
   were also removed since they had no remaining callers.

3. **Quality medium — T1.5 helpers visibility**: The `pub(crate)` modifier on the removed
   `logs_dir_for` and `opener_err_to_app_error` helpers was redundant (same-file only).
   Helpers removed as part of SC-2 cleanup.

4. **Quality low — spawn error `resolved_path`**: Changed spawn `Err` arm from
   `resolved_path: None` to `resolved_path: Some(resolved)`, consistent with all
   other failure arms that have a resolved path available.

5. **Quality low — spawn error log level**: Covered by fix 1; failure arm now uses
   `tracing::warn!` per monitoring spec.

6. **Quality — `commands.test.ts` header**: Updated the file header block comment to list
   all commands tested: T1.1 (getSettings/updateSettings/openLogsFolder) and T1.2 (verifyClaudeCli).

## How the affected component works

### Path resolution

Resolution is a pure, priority-ordered three-step cascade implemented in `resolve_cli_path`:

1. `path_override` argument — caller-supplied absolute path (highest priority).
2. `settings.claude_cli_path` — stored setting read once under the async mutex lock.
3. Bare `"claude"` string — resolved by the OS via `PATH` (lowest priority / fallback).

The helper is `pub(crate)` so it can be called in unit tests without a Tauri runtime.

### Data flow

```
Frontend invoke("verify_claude_cli", { pathOverride })
  → commands::verify_claude_cli(path_override, state)
      → lock state.settings mutex, clone claude_cli_path, drop lock
      → has_override = path_override.is_some()
      → resolve_cli_path(path_override, settings_path)  → PathBuf
      → if has_override: validate_cli_path(&resolved).await
            Err(msg) → warn!(component="cli_detect", kind="cli", path_tried=?resolved, ...)
                     → Ok(CliCheck { found: false, resolved_path: None, error: Some(msg) })
      → tokio::process::Command::new(resolved)
            .arg("--version")
            .stdin(Stdio::null())                        ← prevents stdin blocking
            .stdout(Stdio::piped())
            .stderr(Stdio::piped())
            .spawn()
          Err(_)  → warn!(component="cli_detect", kind="cli", path_tried=?resolved, ...)
                  → CliCheck { found: false, resolved_path: Some(resolved), error: "..." }
          Ok(child)
              → tokio::time::timeout(10s, child.wait_with_output()).await
                  Err(_elapsed) → warn!(component="cli_detect", ...)
                               → CliCheck { found: false, ..., error: "timed out" }
                  Ok(Err(e))    → warn!(component="cli_detect", ...)
                               → CliCheck { found: false, ..., error: "..." }
                  Ok(Ok(output))
                      → raw = stdout || stderr
                      → version = raw.chars().filter(!control).take(256).collect()
                      → info!(component="cli_detect", resolved_path=?resolved, version, mode="interactive", "cli detected")
                      → CliCheck { found: true, resolved_path: Some(resolved), version }
      → Ok(CliCheck)   ← never Err(AppError)
```

### Key contracts

- `verify_claude_cli` always returns `Ok(CliCheck)`. It never propagates `AppError`.
- `found=true` whenever the binary spawns successfully, even if exit code is non-zero.
  This matches CLIs that print version strings to stderr.
- `found=false` only when `spawn()` itself fails (binary not found, permission denied, etc.).
- Logging matches monitoring.md §2.13: `info!` on success (after outcome known), `warn!` on all failure paths.
- A missing CLI (spawn failure) emits `warn!(component="cli_detect", kind="cli", ...)` per §2.13.
  The `CliCheck { found: false }` return value is the soft signal to the frontend; the warn log
  goes to the structured log for diagnosis.
- The settings mutex is locked only long enough to clone `claude_cli_path`, then released
  before the `spawn()` call to avoid holding the lock during I/O.

## Decisions made within Task scope

- `resolve_cli_path` is `pub(crate)` rather than `#[cfg(test)]`-gated because it is
  testable logic with real value outside tests (other future commands may reuse it).
- The settings mutex is dropped before spawning — a scoped block is used to ensure the
  `MutexGuard` is released as soon as the path is cloned, not held across the `await`.
- `found=true` on any successful spawn (including non-zero exit codes). The spec says
  "found=true with version string when CLI is installed"; some CLIs exit non-zero even
  when printing a version banner, so exit code is intentionally ignored.
- Version string prefers stdout over stderr; if both are empty, `version` is `None`.
  A `None` version with `found=true` is valid (unusual but not an error).

// 1. CLI on PATH:
const r1 = await verifyClaudeCli();

### Why `verify_claude_cli` is not integration-tested without a Tauri runtime

`verify_claude_cli` takes `tauri::State<'_, AppState>` which can only be constructed
inside a running Tauri application after `.manage(state)` is called on the builder.
There is no public constructor for `tauri::State` usable from tests.

The spawn/output-capture logic is covered indirectly:
- **Path selection**: IT-01 through IT-08 confirm the correct path is chosen.
- **IPC contract**: FE-T1 through FE-T7 confirm the TS wrapper calls `invoke` correctly.
- **DTO shape**: `bindings_shape.rs` TC-12 confirms `CliCheck` serializes with the right keys.
- **Manual smoke test**: documented in "How to test / verify" section above.

If `verify_cli_spawn_logic` is later extracted as a plain `async fn(PathBuf) -> CliCheck`,
it can be integration-tested directly with a temp script as the probe target.
