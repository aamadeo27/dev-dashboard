# T1.3 — S-01 Setup screen and app-launch CLI check

## What was done

Implemented the full S-01 Setup screen (`src/routes/Setup.tsx`) and wired a CLI-presence check into the Dashboard (`src/routes/Dashboard.tsx`). The Setup screen is shown whenever the Claude CLI is missing from PATH; the Dashboard check enforces this redirect on every fresh app load.

## How the affected components work

### Setup.tsx — data flow

1. On mount, OS is detected once from `navigator.platform` (helper `detectOs()`). No state involved — pure derivation from platform string.
2. The card renders OS-specific install instructions (a `<pre>` code block), a divider, and a path input with Browse + Verify buttons.
3. **Browse**: calls `open({ multiple: false, directory: false })` from `@tauri-apps/plugin-dialog`. If the user picks a file (string returned), it is set as `cliPath` and verify state is reset to `"idle"`.
4. **Verify & Continue**: calls `verifyClaudeCli(cliPath)` (IPC). On `found === true`, calls `updateSettings({ claude_cli_path: cliPath || null })` to persist the path, then sets `verifyState = "success"`. On `found === false` or any rejection, sets `verifyState = "failure"` with an error message. The button is disabled while `verifying === true` or while `cliPath` is empty/whitespace.
5. After success, the "Verify & Continue" button is swapped for a "Continue →" button that calls `navigate("/")` directly (path already persisted).

State variables:
- `cliPath` — controlled input value
- `verifyState`: `"idle" | "verifying" | "success" | "failure"`
- `verifyMessage` — human-readable text for the status area
- `verifying` — boolean guard for disabling UI during in-flight IPC call

### Dashboard.tsx — CLI check effect

A single `useEffect` fires on mount. It calls `verifyClaudeCli(undefined)` (uses the stored setting or PATH fallback). If `result.found === false`, it calls `navigate("/setup", { replace: true })`. IPC rejections are swallowed silently so a transient IPC error never blocks the Dashboard.

## Decisions made within Task scope

- **OS detection via `navigator.platform`**: The spec listed `navigator.platform` and `navigator.userAgent` as candidates. `navigator.platform` produces a shorter, stable string ("Win32", "MacIntel", "Linux x86_64") that matches the spec's "Win" / "Mac" substring checks. No upstream decision needed.
- **`aria-label` on buttons**: Buttons have explicit `aria-label` attributes ("Browse for CLI path", "Verify and continue", "Continue to Dashboard") that override the visible text for accessible name computation. Tests query by `aria-label` accordingly.
- **Verify then persist**: `updateSettings` is only called after `verifyClaudeCli` returns `found: true`. This matches the spec's "on success, persist path" flow and prevents saving a bad path.
- **IPC rejection swallowed in Dashboard**: The spec says "if verify fails entirely (IPC error), don't block the UI." The catch block is intentionally empty.
