# T4.1 — EventParser: Review Findings

## Iteration 1

### Performance Reviewer

**HIGH**

- **H1-perf** `parser.rs:323–327, 343–348, 419–424, 454–459` — Four sites construct `EventParser::new_no_log()` (allocates two `Vec`s) on the hot path just to call one function. Fix: extract free function `dispatch_idle_line(line, events) -> Option<ParserState>`; no throwaway parser needed.

**MEDIUM**

- **M1-perf** `parser.rs:296` — `raw_input.clone()` when `raw_input` is consumed only once in the `is_edit && has_diff_hunk` branch (`diff` field). Move instead.
- **M2-perf** `parser.rs:287–293` — `diff_body: Vec<String>` intermediate allocation filters diff lines only to call `count_diff_lines`. Pass the `input_lines` slice directly to a helper that accepts `&[String]` and skips non-diff prefixes internally.
- **M3-perf** `parser.rs:408` — `to_lowercase()` allocates on every block-end; use `contains` with a case-insensitive matcher or a pre-lowercased constant.

**LOW**

- **L1-perf** — `line_buf` is initialized with `Vec::with_capacity(256)` but never capped; high-volume output will cause repeated reallocations. Addressed by security FIX-1 (cap enforces size).

### Security Reviewer

**CRITICAL**

- **C1-sec** `parser.rs:122` — `self.line_buf.push(byte)` has no upper bound. An actor streaming a single line with no `\n` can grow `line_buf` without limit (heap exhaustion / OOM). **Fix**: add `const MAX_LINE_BYTES: usize = 1_048_576` (1 MiB); when `line_buf.len() >= MAX_LINE_BYTES`, flush as System event and reset.
- **C2-sec** `parser.rs:354, 389, 429, 464` — All four block accumulators (`input_lines`, `lines` × 3, `diff_lines`) grow without bound. **Fix**: add `const MAX_BLOCK_LINES: usize = 10_000` and `const MAX_BLOCK_BYTES: usize = 4_194_304` (4 MiB); when exceeded, emit System event with truncation notice and transition to Idle.

**HIGH**

- **H1-sec** `parser.rs:307–309` — `serde_json::from_str(&raw_input)` has no size gate; a 100 MiB input string would be allocated before the parse attempt. **Fix**: gate `raw_input.len()` against `MAX_BLOCK_BYTES` before calling `from_str`.
- **H2-sec** `parser.rs:408` — `is_error` determined by `raw_output.to_lowercase().contains("error")`; attacker-controlled output can inject the word "error" to mark a successful result as failed, or vice versa for an actual error by never including the word. Flag only: no deterministic fix without a structured result format (deferred to T4.3 caller).
- **H3-sec** `parser.rs:251, 298–303, 374, 382–386, 402–406, 446–451` — Attacker-controlled stdout is stored verbatim in `RunEvent` string fields (`AssistantText.text`, `Thinking.text`, `ToolResult.output` string variant, `System.text`). Fields may contain ANSI escape sequences, C0/C1 control characters that affect terminal rendering or log parsing. **Fix**: strip control chars (C0 range 0x00–0x1F except 0x09/0x0A/0x0D, and C1 0x80–0x9F) from all string fields before building events.
- **H4-sec** `parser.rs:299` — `FileEdit.path` is set to `tool_name` ("Edit" or "Write"), not the actual file path. Downstream consumers (T4.3/T4.7) relying on `path` for access control or audit will see only the tool name — a security control bypass. **Fix**: parse path from `--- a/<path>` or `+++ b/<path>` diff header lines.

**MEDIUM**

- **M1-sec** `parser.rs:333, 411` — `uuid::Uuid::new_v4()` generates random, unordered IDs; adjacent events cannot be correlated by ID sequence or sorted by creation time. Use `uuid::Uuid::now_v7()` (time-ordered) so events are auditable and trace-able. Breaks call_id ↔ ToolCall.id correlation if v4 IDs are already persisted.
- **M2-sec** — No maximum on the number of events returned per `feed()` call; malicious output can cause unbounded event vec growth within a single call. Addressed indirectly by C2 accumulator caps; no separate fix needed.

**LOW**

- **L1-sec** — `STEP_FAILED_SENTINEL` is empty string; any future assignment without a corresponding test update could silently match every line. Current behaviour is safe; flag for T4.8.

### Scope Reviewer

**HIGH**

KB §6.7 does not specify a sentinel for user input lines, so this may be intentional — but the omission is undocumented. **Action**: add a comment in `process_idle_line` noting that `UserInput` is not emitted (CLI stdout does not echo user input; variant is populated by T4.3 caller).
- **H2-scope** `parser.rs:77–86` — KB §7.3/§7.4 requires per-event monitoring counters (`events_parsed_total`, `parse_errors_total`, `block_truncations_total`). No `metrics::counter!` or `tracing::increment_counter!` calls present. **Action**: add tracing spans or counter increments at event-emit sites. Defer full metrics wiring to T9.2 but add placeholders now.
- **H3-scope** `parser.rs:333, 411` — UUID v4 used. Monitoring spec (KB §7) requires time-ordered IDs for event correlation. Must be v7. (Same as security M1 — converging fix.)

**MEDIUM**

- **M1-scope** — NFR-5 (no panic on any input) is tested indirectly through the invalid-UTF-8 and malformed-block tests but not by a dedicated fuzz-style harness. Record as open gap; defer to T10 (test-backfill).
- **M2-scope** — Throwaway re-dispatch pattern (`new_no_log()` + `process_idle_line`) was a coder decision with significant behavioral implications (state dropped, adjacent-tool-call events lost). **Escalate to Architect** before T4.3 consumes parser output.

### Code Quality Reviewer

**HIGH**

- **H1-quality** `parser.rs:323–327, 343–348, 419–424, 454–459` — `new_no_log()` creates a throwaway `EventParser`, calls `process_idle_line` on it, captures events, then **discards the parser's new state**. If the triggering line is itself a block opener (e.g., `⏺ Tool: Beta`), the new `AccumulatingToolCall` state is lost — Beta ToolCall is never emitted. This is a real data-loss bug, not just dead code. **Fix**: extract free function `fn dispatch_idle_line(line: &str, events: &mut Vec<RunEvent>) -> Option<ParserState>` and return its result from the static handler so the caller sets `self.state` correctly.
- **H2-quality** `parser.rs:299` — `path: tool_name.clone()` stores "Edit" or "Write" instead of the actual edited file path. Test `edit_tool_with_diff_emits_file_edit_event` doesn't assert `path`, masking this bug. **Fix**: parse path from `--- a/<path>` diff header in accumulated `input_lines`.

**MEDIUM**

- **M1-quality** `parser.rs:266, 381, 399, 440` — `line.starts_with(' ') || line.starts_with('\t')` duplicated at four sites. **Fix**: extract `#[inline] fn is_indented(line: &str) -> bool`.
- **M2-quality** `parser.rs:296` — `raw_input.clone()` to populate `diff` when `raw_input` is never used after that point in the branch. Move instead.
- **M3-quality** `parser.rs:408` — `is_error` heuristic (`contains("error")`) is undocumented and surprising. Add a comment: `// heuristic — structured error flag awaits T4.3 protocol`.
- **M4-quality** `parser.rs:433–466` — `AccumulatingDiff` state handler (`process_diff_line`) exists but the state is never transitioned into. `process_tool_call_line` handles diff detection inline and emits `FileEdit` directly. The dead state handler is misleading and adds maintenance surface. **Fix**: remove `AccumulatingDiff` variant and `process_diff_line`; the inline path already handles it.
- **M5-quality** `parser.rs:383` — `process_thinking_line` does not re-dispatch the closing non-indented line (alt-marker variant). Closing line is silently dropped. Test documents this as "current behaviour" but it is a loss of potentially significant content. Low-priority, document explicitly.

**LOW**

- **L1-quality** — Test `adjacent_tool_calls_no_blank_line_between` asserts broken behaviour (Beta dropped). After FIX-4 (dispatch_idle_line returning state), Beta will be emitted. Test must be updated to assert both Alpha and Beta.
- **L2-quality** — `process_thinking_line` takes `needs_close_tag: bool` as a copied parameter; it could borrow from the enum variant. Minor ergonomic issue.
- **L3-quality** — `EventParser::default()` calls `new()` which emits a tracing log line; constructing a default parser in a tight loop would spam logs. `new_no_log()` is the right constructor for non-production use but is private. Consider renaming `new()` → `new_logging()` and making `new_no_log()` → `new()` (or `Default`). Out of T4.1 scope; escalate to Architect.

## Iteration 2 (re-review after fix pass)

**Perf**: No CRITICAL/HIGH. MEDIUMs still open: `diff_body` intermediate `Vec<String>` in `process_tool_call_line` (can be replaced with a single fold over `input_lines`); `to_lowercase()` alloc on every ToolResult close (use `eq_ignore_ascii_case` byte-window scan). LOWs: O(n) `total_bytes` recomputed per accumulated line (add running counter to accumulator variants); `dispatch_idle_line` allocates a dummy `EventParser` per redirect (extract `process_idle_line` as free fn).

**Security**: No CRITICAL/HIGH. MEDIUMs: `extract_diff_path` returns attacker-controlled path with no sanitization — no control-char strip, no traversal (`..`) check, no length cap (downstream consumers inherit raw value); `strip_control_chars` misses DEL (U+007F) and bidi/Trojan-Source characters (U+200E/F, U+202A–U+202E, U+2066–U+2069, U+2028/9). LOWs: `MAX_LINE_BYTES` confirmed safe (informational only); `serde_json` depth not capped (nested `[[[[…]]]]` within size cap can overflow stack); `dispatch_idle_line` allocation (same as perf LOW).

**Code quality**: No CRITICAL/HIGH. MEDIUM: `is_error` heuristic at line 519 still has no comment — prior fix pass note was not implemented. LOWs: `dispatch_idle_line` allocates unused `line_buf` Vec in dummy parser (same as perf LOW); `AccumulatingDiff` row still present in T4.1.md state-machine table (stale after FIX-10); `process_idle_line` has `&mut self` receiver that never mutates self — add in-code comment explaining why.

**Verdict**: All three reviewers report no CRITICAL or HIGH findings. T4.1 review+fix loop **DONE**.

## Escalations

- **Architect**: Throwaway re-dispatch pattern — coder decision to create a dummy parser for line redirect; after FIX-4 this is replaced by a cleaner approach. Acknowledge and close, or redesign.
- **Architect**: `new()` vs `new_no_log()` visibility — logging constructor vs. silent constructor naming; consider which should be `Default`.
- **T4.3 caller**: `is_error` heuristic is unstructured; T4.3 should supply a structured `is_error` flag via protocol, not rely on keyword scan.
- **T4.8**: Set `STEP_FAILED_SENTINEL` to the real marker before T4.8 ships; tests already guard the empty-sentinel invariant.
