# T0.7 — Local Dev Infrastructure

## What was done

Set up dev-only local infrastructure for the Dev Dashboard project: a gitignored `.env.local` file for dev-specific environment variables, an isolated `.dev-data/` config directory, a stub `logging.rs` Rust module wired into `lib.rs`, and a frontend `env.ts` utility.

## How the affected components work

### `.env.local`

Vite automatically gitignores `*.local` env files. When running `pnpm dev`, Vite loads `.env.local` and exposes its values via `import.meta.env` (for `VITE_`-prefixed vars) or as process-level env vars for the Rust side. The two vars set here:

- `DEV_DASHBOARD_LOG=debug` — read by `init_logging()` via `EnvFilter::try_from_env("DEV_DASHBOARD_LOG")` to set `tracing` log level.
- `DEV_DASHBOARD_CONFIG_DIR=.dev-data` — intended for future use by `SettingsStore` / `ProjectRegistry` to override the OS config dir, keeping dev data out of the real user config. The Rust side does not yet consume it (T0.6 TODO in `logging.rs`).

### `.dev-data/`

An empty directory tracked via `.gitkeep`. Purpose: isolated config root for dev runs so that dev activity doesn't pollute `%APPDATA%\dev-dashboard\` (or OS equivalent). The `.gitignore` already had `.dev-data/` in the Dev Dashboard-specific section; no change was needed there.

### `src-tauri/src/logging.rs`

A stub module. The public function `init_logging(_config_dir: &Path)` sets up a basic `tracing-subscriber` `fmt` layer writing to stderr, filtered by the `DEV_DASHBOARD_LOG` environment variable (defaults to `info` if unset or invalid). The `_config_dir` parameter is reserved for T0.6, which will wire in `tracing-appender` daily file rotation writing JSON-formatted lines to `<config_dir>/dev-dashboard/logs/app-YYYY-MM-DD.log` (KB §7.2).

The module is declared as `pub mod logging;` in `lib.rs` so it is compiled as part of the library crate and available to `main.rs` via `dev_dashboard_lib::logging::init_logging(...)`.

### `src/utils/env.ts`

Single exported constant:

```typescript
export const isDev = import.meta.env.DEV;
```

`import.meta.env.DEV` is a boolean injected by Vite at build time: `true` in `vite dev`, `false` in production builds. Consumers import `isDev` from this module rather than reading `import.meta.env.DEV` inline, so the abstraction point is in one place.

## Decisions made within Task scope

- Inserted `pub mod logging;` in alphabetical order between `ipc` and `projects` in `lib.rs` to match the existing module ordering convention.
- `env.ts` uses a `const` export (not `default`) to align with the existing `format.ts` / `markdown.ts` convention in `src/utils/`.

```
pnpm test

```

All three JS checks pass clean. The Rust stub compiles because `tracing-subscriber` with `env-filter` feature is already declared in `src-tauri/Cargo.toml`.
