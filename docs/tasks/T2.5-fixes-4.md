# T2.5-fixes-4 — Skeleton animation reduced-motion guard

## What was done

Added a module-level `skeletonAnimation` const that conditionally omits the `animation` property when `reducedMotion` is true. The four skeleton style entries (`skeletonName`, `skeletonBtn`, `skeletonGit`, `skeletonMeta`) previously hardcoded `animation: "pc-skeleton 1.5s ease-in-out infinite"` unconditionally. They now spread `...skeletonAnimation` instead.

## How the affected component works

`ProjectCard.tsx` uses inline `style=` props for all skeleton elements. The existing `@media (prefers-reduced-motion: reduce)` block in `ensurePulseKeyframes` targets CSS class names (`.pc-skeleton`), which never match inline styles — so that CSS rule was dead code for skeleton elements. The correct mechanism is the JS-side `reducedMotion` boolean evaluated once at module load via `window.matchMedia`. This pattern was already used for `runningBadge` and `OUTCOME_STYLES[Running]`.

`skeletonAnimation` is computed once when the module first loads. All four skeleton entries spread it, so there are zero per-render allocations and no conditional logic inside the `styles` object literal.
