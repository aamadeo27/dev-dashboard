# T2.4 — Frontend hooks (useProjects, useGitStatus, useVisibleProjects)

## What was done

Added a typed Tauri event `subscribe` helper to the IPC events module, then implemented three hook files that connect the React UI to the Tauri backend: a TanStack Query hook for fetching projects, a Zustand-backed hook set for git status (listener, selector, and viewport reporting).

## How the affected components work

### `src/ipc/events.ts` — subscribe helper

`subscribe<T>(eventName, handler)` wraps `@tauri-apps/api/event`'s `listen`. It registers the handler against the Tauri event bus and returns the `UnlistenFn` promise so callers can clean up. The generic `T` flows to `event.payload`, giving callers full type safety without additional casts.

### `src/hooks/useProjects.ts` — TanStack Query wrapper

`useProjects()` calls `useQuery` with `queryFn: listProjects` (the Tauri command wrapper from `commands.ts`). The query key `PROJECTS_QUERY_KEY = ["projects"]` is exported so other call sites can target the same cache entry for invalidation. The returned shape normalises `data` to `[]` when undefined (initial load) and casts TanStack's `Error | null` union to `Error | null`.

### `src/hooks/useGitStatus.ts` — Zustand store + hooks

**Store**: `useGitStatusStore` holds `statuses: Record<string, GitStatus>` keyed by project id. `setStatus` is a shallow-merge updater so unrelated project entries are never clobbered.

**`useGitStatusListener()`**: Mount once at the app root (e.g. `App.tsx`). Subscribes to `git:updated` events from the backend, extracts `{ id, status }` from the payload, and calls `useGitStatusStore.getState().setStatus(id, status)` directly — bypassing React render to avoid triggering global re-renders on every poll tick. Cleanup unlistens on unmount.

**`useGitStatus(id)`**: Selector hook. Returns `GitStatus | null` for a single project id. Only re-renders the subscriber when that project's entry changes.

**`useVisibleProjects(ids)`**: Keeps the backend git poller in sync with the UI viewport.
- Debounced (300 ms) call to `setVisibleProjects(ids)` on `ids` content changes; `JSON.stringify(ids)` is the dep key to avoid referential instability from parent re-renders producing new array references.
- `window "blur"` calls `setVisibleProjects([])` immediately to pause polling when the window loses focus.
- `window "focus"` calls `setVisibleProjects(idsRef.current)` immediately to resume with the latest ids without needing to re-register the event listener.
- `idsRef` tracks the latest `ids` value so the stable focus handler always has current data.

## Decisions made within Task scope

- The `import` statement for `@tauri-apps/api/event` was placed at the top of `events.ts` (before the `export const` declarations) rather than inline at the bottom, as hoisting imports to the file top is the TypeScript/ESLint convention and avoids a parse-order warning.

## How to test / verify

Run the TypeScript compiler to confirm no type errors:

```
pnpm tsc --noEmit
```

Run the unit test suite:

```
pnpm test
```

Manual smoke-test sequence:
1. Launch the app (`pnpm tauri dev`).
2. Open DevTools console — `useGitStatusListener` should register without errors.
3. Add a git project; `useProjects` should return it on next render.
4. Scroll a project card into view; the backend poller should start emitting `git:updated` events visible in the console if you add a temporary `console.log` in the listener.
5. Alt-Tab away from the window; `set_visible_projects` with `[]` should fire (verifiable via Tauri log).
