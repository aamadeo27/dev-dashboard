# T1.3-fixes-2 — Setup Screen Quality/Perf Fixes (Round 2)

## What was done

Applied 5 targeted quality and performance fixes to `Setup.tsx` and `Setup.test.tsx` following the iteration-2 review of T1.3. No behaviour changed; all 168 tests continue to pass.

## How the affected component works

`Setup.tsx` is the S-01 onboarding screen. It renders a card with install instructions, a path input, Browse (opens a file dialog), and Verify & Continue. State is a `verifyState` discriminated union (`idle | verifying | success | failure`). All styling is via a module-level `styles` constant of `Record<string, React.CSSProperties>` using CSS variable tokens.

The fixes in this task address:
- **Per-render object allocation**: spread expressions like `{ ...styles.a, ...styles.b }` in JSX create a new object on every render. Hoisting the combined form into the `styles` constant makes it a single stable object reference.
- **Duplicate string literals**: the `pathCheckError` message was constructed identically in two `catch` branches.
- **Dead alias**: `inputDisabled` was a one-liner alias for `isVerifying` with no extra logic.
- **Test quality**: one test asserted only the shape of a local constant it constructed itself (tautology), another mock was set up for a code path that is never reached when `found: false`.

## Changes detail

### FIX 1 — Hoist inline style spreads

Added 6 new entries to the `styles` constant at the bottom of `Setup.tsx`:

- `statusVerifying` — replaces `{ ...styles.statusText, color: "var(--text-secondary)" }`
- `statusSuccess` — replaces `{ ...styles.statusText, color: "var(--success)" }`
- `statusFailure` — replaces `{ ...styles.statusText, color: "var(--error)" }`
- `inputDisabledFull` — replaces `{ ...styles.input, ...styles.inputDisabled }`
- `secondaryDisabledFull` — replaces `{ ...styles.secondaryButton, ...styles.buttonDisabled }`
- `primaryDisabledFull` — replaces `{ ...styles.primaryButton, ...styles.primaryButtonDisabled }`

Updated `renderStatusArea()` and three JSX elements in the return to use the new keys.

### FIX 2 — Extract duplicate error string

Added `const pathCheckError = \`Could not run \\\`${cliPath} --version\\\`. Check the path.\`` at the top of `handleVerify`. Both the `else` branch (`result.found === false`) and the outer `catch` now reference `pathCheckError`.

### FIX 3 — Remove `inputDisabled` alias

Removed `const inputDisabled = isVerifying;` and its preceding comment. All three JSX consumers (input `disabled`, input `style`, Browse button `disabled`/`style`) now reference `isVerifying` directly. No other references existed.

### FIX 4 — Replace tautology test

The test "passes Windows executable filters when OS is windows" constructed a local object and then asserted properties of that same local object — it never touched the component under test. Replaced with an explanatory comment describing why this cannot be tested via `open()` call-args in jsdom (module-level `OS` constant is fixed at import time).

### FIX 5 — Remove dead `updateSettings` mock

In the "disables input and Browse while verifying" test, `resolveVerify` is called with `found: false`. The `handleVerify` code path for `found === false` never calls `updateSettings`. The `vi.mocked(updateSettings).mockResolvedValue(...)` block that followed was dead. Removed it.
