# T2.2 ProjectScanner — Review Fixes

## What was done

Five review findings were resolved in the ProjectScanner subsystem. No behavioral
changes — the scanner detection logic is identical. Changes are confined to
correctness of async usage, comment accuracy, test hygiene, and code style.

## How the affected component works

`ProjectRegistry` (in `src-tauri/src/projects/mod.rs`) is an in-memory store
backed by `projects.json`. All mutations go through async methods on the struct
while holding a Tokio mutex from `AppState`. The `scanner::detect` function
(in `src-tauri/src/projects/scanner.rs`) performs up to 5 synchronous
`Path::exists()` stat syscalls to identify language/package-manager.

The IPC layer (`src-tauri/src/ipc/commands.rs`) exposes Tauri commands that
acquire the registry mutex, delegate to `ProjectRegistry` methods, and return.

## Decisions made within Task scope

- In `scan_project`, the project is looked up twice (once to clone the path
  before `spawn_blocking`, once to mutate after). This is required because a
  `&mut` reference cannot be held across an `await` point. An `id.to_owned()`
  captures the id for the second lookup. This is functionally identical to the
  original single-lookup version.
- The `scanner::detect` doc comment at the top of `scanner.rs` says
  "called outside the async mutex and is acceptable for blocking use on the
  Tokio thread pool" — this is now accurate because FIX-1 ensures the call
  happens inside `spawn_blocking`, keeping it off the async executor.

## Fix details

### FIX-1 — spawn_blocking in add_project and scan_project

`scanner::detect` makes synchronous stat calls. Calling it directly on the
Tokio executor thread blocks the runtime. Both `add_project` and `scan_project`
now clone the path first, call `tokio::task::spawn_blocking`, and await the
result before any mutable borrow on `self`.

In `scan_project`, the project is found by id to get the path, the await
happens, then the project is found again by id to apply the updated fields.

### FIX-2 — comment in canonicalize_dir

The comment `// Verify it is a directory via metadata (single async stat)` was
wrong: `canonicalize` is itself one async call, making the metadata check the
second. The comment now reads "second async call after canonicalize".

### FIX-3 — duplicate test removed

`detect_npm_priority_when_no_pnpm_lock` was identical in setup and assertions
to `detect_typescript_npm_without_pnpm_lock`. The duplicate was removed;
the retained test already has a comment noting "No pnpm-lock.yaml — should
fall back to npm."

### FIX-4 — doc comment on detect_pyproject_alone_returns_none

Added a three-line doc comment explaining that Python detection intentionally
requires a lock file to avoid false-positives on non-Python projects that
happen to include a `pyproject.toml`.

### FIX-5 — fully-qualified paths in scan_project command

The `scan_project` Tauri command used `crate::app_state::AppState`,
`crate::error::AppResult`, and `crate::projects::Project` when `AppState`,
`AppResult`, and `Project` are already in scope via the `use` declarations at
the top of `commands.rs`.
