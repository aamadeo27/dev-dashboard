# T2.1 ‚Äî ProjectRegistry CRUD

## What was done

Implemented `ProjectRegistry` ‚Äî the in-memory, file-backed store for registered
projects. Wired it into `AppState`, added six Tauri commands, added five
TypeScript wrappers (not `renameProject`, which is internal-only per KB ¬ß5.1),
and wrote unit tests at both the Rust and TypeScript layers.

## How the affected component works

### Data model

`Project` (already defined in `projects/mod.rs`) is the persisted and IPC
type. All fields are serialized to `projects.json` except `is_missing`, which is
computed live on every `list_projects()` call via `path.exists()`.

### ProjectRegistry

Holds `Vec<Project>` in memory and a `PathBuf` pointing to the config directory.

**Load** (`ProjectRegistry::load(config_dir)`):
- Reads `<config_dir>/projects.json`.
- Missing file ‚Üí empty registry (normal first-run).
- Parse error ‚Üí `tracing::warn!` + empty registry (avoids startup crash).

**Save** (`save(&self) -> AppResult<()>`):
- Serializes `projects` to pretty JSON.
- Writes to `<config_dir>/projects.tmp`.
- Renames to `<config_dir>/projects.json` (atomic on POSIX; best-effort on
  Windows where rename fails if the target is locked).

**CRUD operations** (all synchronous, holding an `&mut self` borrow):

| Method | Errors |
|---|---|
| `add_project(path)` | `InvalidInput` if path not found/not a dir; `AlreadyExists` if canonical path already registered |
| `remove_project(id)` | `NotFound` |
| `relocate_project(id, new_path)` | `NotFound`, `AlreadyExists` if new_path taken by another project |
| `set_project_tags(id, tags)` | `NotFound` |
| `rename_project(id, name)` | `NotFound` |
| `list_projects()` | never errors; computes `is_missing` live |

Tag normalization (`normalize_tags`): trim, lowercase, deduplicate preserving
first-occurrence order, drop empty strings.

### AppState integration

`projects: Arc<tokio::sync::Mutex<ProjectRegistry>>` added to `AppState`.
`ProjectRegistry::load(&config_dir)` is called once during `run()`, before
`AppState` is constructed. Each command handler locks the mutex, calls the
synchronous registry method, and returns.

### IPC commands (Tauri)

Six `#[tauri::command]` functions in `ipc/commands.rs` ‚Äî thin shims that
acquire the mutex and delegate to the registry:

```
list_projects()          -> Vec<Project>
add_project(path)        -> Project
remove_project(id)       -> ()
relocate_project(id, new_path) -> Project
set_project_tags(id, tags)     -> Project
rename_project(id, name)       -> Project   (no TS wrapper)
```

All six are registered in `lib.rs` in both `#[cfg(debug_assertions)]` and
`#[cfg(not(debug_assertions))]` blocks.

### TypeScript wrappers

Five wrappers in `src/ipc/commands.ts` (camelCase):

```ts
listProjects(): Promise<Project[]>
addProject(path: string): Promise<Project>
removeProject(id: string): Promise<void>
relocateProject(id: string, newPath: string): Promise<Project>
setProjectTags(id: string, tags: string[]): Promise<Project>
```

`renameProject` is intentionally absent ‚Äî it is an internal Rust command per
KB ¬ß5.1 note.

## Decisions made within Task scope

- `save()` is synchronous (`std::fs`) rather than async (`tokio::fs`). The
  registry mutex is held for the duration, and all callers are async command
  handlers; blocking the executor for a small JSON write (typically < 1 ms) is
  acceptable and avoids the complexity of making `ProjectRegistry` aware of the
  async runtime. If write latency becomes measurable this can be moved to
  `tokio::task::spawn_blocking` without API changes.
- `normalize_tags` is a `pub` (not `pub(crate)`) associated function so that
  unit tests can call it directly in the `#[cfg(test)]` module.
- `relocate_project`: if the project is relocated to its own current canonical
  path (same id), the duplicate check passes (predicate requires `p.id != id`)
  and the path is updated (effectively a no-op on path but `last_modified` is
  set). This is the least-surprising behavior.

## How to test / verify

### Rust unit tests (cargo required)

```sh
cargo test -p dev-dashboard-lib projects::tests
```

Tests cover:
- `add_project` ‚Üí returns project, `is_missing=false`, persists across reload.
- Duplicate path ‚Üí `AlreadyExists`.
- File path / nonexistent path ‚Üí `InvalidInput`.
- `remove_project` ‚Üí removes from list; non-existent id ‚Üí `NotFound`.
- `set_project_tags` ‚Üí lowercases, trims, deduplicates; non-existent ‚Üí `NotFound`.
- `is_missing=true` after directory deleted.
- `relocate_project` ‚Üí updates path + `last_modified`; duplicate target ‚Üí `AlreadyExists`; unknown id ‚Üí `NotFound`.
- `rename_project` ‚Üí updates name + `last_modified`; unknown id ‚Üí `NotFound`.
- Edge: empty tags, whitespace-only tags.

### TypeScript unit tests

```sh
pnpm test --run src/ipc/commands.test.ts
```

24 new tests (48 total in file) covering all 5 wrappers:
correct command name, argument keys, return value pass-through, error
propagation, and that `renameProject` is not exported.

### Manual smoke test

Build and run the Tauri app, then from browser DevTools:
```js
await window.__TAURI__.core.invoke("list_projects")        // []
await window.__TAURI__.core.invoke("add_project", { path: "/some/existing/dir" })
await window.__TAURI__.core.invoke("list_projects")        // [{ id, name, ... }]
```
