# T2.3-fixes — GitPoller Review Fixes

## What was done

Applied 8 review findings to the T2.3 GitPoller implementation. Changes cover performance (parallel dispatch), security (input sanitization and size cap), and code quality (struct placement, Default impl, comments, test logic).

## How the affected component works

`GitPoller` is a background Tauri task (`git.rs`) that periodically polls git status for visible projects and caches results in a shared `HashMap`. The poll loop:

1. Reads `git_poll_interval_secs` from settings once per cycle.
2. Sleeps in 500 ms ticks, counting only unpaused time; resets the counter on window blur.
3. Snapshots visible project IDs from `poller.visible`.
4. Acquires the `projects` mutex once, collects all `(id, path)` pairs.
5. Dispatches all `git_status_for_path` calls concurrently via `tokio::task::JoinSet::spawn_blocking`.
6. Drains results, acquires the statuses cache mutex once, updates the cache, and emits `git:updated` events.

The `set_visible_projects` IPC command replaces the entire visible set and now caps input at 1024 IDs. The `get_git_status` command returns a cached `GitStatus` by project ID.

## Decisions made within Task scope

- FIX-5: The variable `paused_for_event` is already `Arc<AtomicBool>` (a clone of `poller.paused`), so the comment was added above the existing `.store(...)` call without renaming the variable. The spec's `poller_for_event.paused.store(...)` was illustrative of the intended comment, not a rename directive.
- FIX-2: `error_msg[..512]` and `b[..256]` slice on byte boundaries. Both `git2` error messages and branch short names are ASCII in practice; the spec's stated approach was followed exactly.
- FIX-8: `crate::projects::git::GitStatus` is kept as a qualified path in `commands.rs` (no new `use` import added) since no other type from `crate::projects::git` is imported in that file — this is consistent with how the file already handled it.
