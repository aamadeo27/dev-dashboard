# T2.7 — Project card context menu, tag editor popover, and remove confirmation

## What was done

Replaced two stub components (`ContextMenu`, `TagEditorPopover`) with full implementations and wired them into `Dashboard.tsx` alongside a per-card remove confirmation overlay.

## How the affected components work

### ContextMenu

Fixed-position overlay that renders a list of action buttons at a given `(x, y)` viewport coordinate. Closes on Escape keydown or mousedown outside `[data-context-menu]`. Items can be marked `danger` (red text) or `disabled` (greyed out, not clickable). Each button calls `item.onClick()` then `onClose()`.

### TagEditorPopover

Fixed-position overlay for editing a project's tags. Renders current tags as removable chips and an input to add new ones. On every add or remove it calls `setProjectTags` IPC, then calls `onTagsChange(projectId, newTags)` so the parent can invalidate the React Query cache. Closes on Escape or mousedown outside `[data-tag-editor]`. The input auto-focuses on mount. Tags are trimmed, lowercased, and deduplicated before being passed to the IPC call.

### Dashboard — context menu wiring

Three new state slices:
- `contextMenu` — stores the project and screen coordinates when a card is right-clicked.
- `tagEditorProject` — set when "Edit Tags" is chosen from the context menu.
- `confirmRemoveId` — set when "Remove" is chosen; the card at that id renders a confirmation overlay instead of the normal card.

`handleContextMenu(e, project)` calls `e.preventDefault()` and sets `contextMenu`.

`buildMenuItems(project)` is a plain function defined inside the Dashboard component body. It closes over `contextMenu`, `queryClient`, and state setters. For missing projects it adds a "Relocate" item. All items are typed via the inline `ContextMenuItem` type.

`handleTagsChange` invalidates the `PROJECTS_QUERY_KEY` query on every tag change.

`ProjectCardWrapper` gained five new props (`confirmingRemove`, `onContextMenu`, `onRelocate`, `onRemove`, `onConfirmRemove`, `onCancelRemove`). When `confirmingRemove` is true it renders an in-place confirmation card (red border, Remove + Cancel buttons) instead of the normal `ProjectCard`.

The two overlays (`ContextMenu`, `TagEditorPopover`) are rendered at the bottom of the root `<div>` so they sit above all card content via `zIndex: 1000/1001`.

## Decisions made within Task scope

- `buildMenuItems` is a plain function (not `useCallback`) per spec — no memoization needed since it only runs when the context menu opens.
- The `ContextMenuItem` type is defined inline in `Dashboard.tsx` (not re-exported from `ContextMenu.tsx`) per spec.
- Overlays are rendered inside the root `<div>` (not a portal) because the root `<div>` is `position: relative` and the overlays use `position: fixed`, so they escape normal flow without needing a portal.
- `setProjectTags` errors are silently swallowed in `TagEditorPopover`; the backend emits a `toast:show` event on failure per KB §5.
