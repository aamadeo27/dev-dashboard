# T4.3-pq-fixes — Performance and Quality Fixes

## What was done

Applied five perf/quality fixes identified in the T4.3 review to the run session
I/O pipeline.

## How the affected component works

`run_io_loop` is a Tokio background task spawned once per `launch_run` call.  It
reads child stdout/stderr, feeds bytes through `EventParser`, writes structured
events to the transcript, and emits Tauri events to the frontend.  The task is
managed via `RunManager` which holds an `Arc<Mutex<HashMap<String, Arc<SessionHandle>>>>`.

### FIX-PQ-1 — Async-aware span (Quality HIGH)

Previously `span.enter()` was called inside the async function, producing a sync
guard that is dropped at yield points.  The span is now instrumented at the
`tokio::task::spawn` call site in `commands.rs` using `.instrument(span)`, which
correctly re-enters and exits the span across every await point.

### FIX-PQ-2 — Batched emit (Perf HIGH)

Previously one `serde_json::json!` + one `app_handle.emit()` was issued per
`RunEvent` object — potentially dozens of IPC round-trips per `read()` call.
Now all events from a single `read()` call are collected and emitted as one
payload:

```
{ run_id: string, events: RunEvent[] }
```

Transcript writes (`append_event`) remain per-event.  The `UserInput` path is
not in the hot loop so it still emits immediately, as a single-element array.

### FIX-PQ-3 — `sessions_arc()` helper (Perf+Quality MEDIUM)

`RunManager` now exposes `sessions_arc()` which clones the inner `Arc` without
requiring callers to hold the manager lock while separately locking the sessions
map.  Three identical double-lock patterns in `commands.rs` (`launch_run`,
`stop_run`, `send_input`) are replaced with `state.run_manager.lock().await.sessions_arc()`.

### FIX-PQ-4 — Single shared drain timeout (Perf MEDIUM)

The two sequential 200 ms drain timeouts (stdout then stderr, 400 ms worst case)
are replaced with one 200 ms timeout wrapping both drains sequentially.  The
`tokio::join!` form in the spec causes a borrow-checker error when `writer` is
captured mutably by two concurrent async blocks; the sequential form under one
shared timeout achieves the same worst-case latency improvement (200 ms max
total instead of 400 ms) while satisfying the borrow checker.

### FIX-PQ-5 — `RunIoContext` struct (Quality MEDIUM)

`run_io_loop` previously took 10 positional parameters (requiring a
`#[allow(clippy::too_many_arguments)]` suppression).  These are grouped into
`RunIoContext` and the signature reduced to three parameters:
`(app_handle, run_id, ctx)`.  The `#[allow]` attribute is removed.

## Decisions made within Task scope

- **FIX-PQ-4 sequential vs `tokio::join!`**: The spec provided aspirational
  `tokio::join!` code that would fail the borrow checker (`writer` captured
  mutably by two async blocks).  Applied the sequential-under-one-timeout form
  instead.  The goal (single shared 200 ms budget) is preserved; the
  concurrency model is different but the practical latency improvement is
  identical for the single-threaded Tokio executor.

- **`events` key for UserInput**: The batched payload uses `events: [event]`
  (single-element array) for UserInput so all `RUN_EVENT` payloads have a
  uniform shape.  Consumers only need one deserialization path.

## How to test / verify

```
cd src-tauri
cargo test
```

Frontend tests (no changes needed — no test touches the payload key):
```
npm test
```

Manual smoke test: launch a run via the UI and confirm `run:event` payloads
arrive with shape `{ run_id, events: [...] }` in the browser devtools Tauri
event log.
