# T4.8 — Research: Step-Failure Interaction Protocol

**Epic**: 4 — Run Execution Core  
**Type**: Research/Spike (backend)  
**Branch**: `feat/T4.8-step-failure-research`

## Goal

Before T4.7 can be implemented, determine how the Claude CLI interactive mode handles step-failure mid-run. Two possible protocols:

- **(a) Stdin tokens**: Claude CLI waits and reads specific stdin tokens to choose Retry / Skip / Abort / Continue.
- **(b) Kill + re-invoke**: No stdin protocol; each choice is implemented by killing the subprocess and optionally re-invoking with modified args.

2. Read `src-tauri/src/runs/session.rs` (RunSession, T4.3 impl) to understand current stdin-write path.
3. Research the Claude CLI step-failure behavior:

   - Search for any documented stdin protocol (e.g., `y\n`, `n\n`, `s\n` tokens, or special escape sequences).
   - Check what `EventParser` currently emits as a `StepFailed` event (step-failure marker patterns).
4. Document the concrete finding in `docs/kb/system-design.md` §9 item 3 (BLOCKER-02).
5. Update T4.7 acceptance criteria in `docs/epics/epic-4-run-execution.md` to match the resolved protocol.

## Findings (iteration 1 — 2026-05-24)

### Protocol resolved: option (b) kill + re-invoke

No evidence of a stdin token protocol was found in any source examined. The conservative default (b) was selected.

### Evidence reviewed

| Source | Finding |
|---|---|
| `docs/tasks/T1.2.md` | T1.2 probe ran `claude --version` only, with `.stdin(Stdio::null())`. No step-failure prompts, interactive stdin behavior, or `--help` output was captured. |
| `src-tauri/src/runs/parser/patterns.rs` | `STEP_FAILED_SENTINEL = ""` — deliberately disabled. No "waiting for input" marker was ever identified in CLI output patterns. TODO comment says T4.8 will set the real value. |
| `src-tauri/src/runs/parser.rs` | `process_line` checks `!STEP_FAILED_SENTINEL.is_empty()` before firing `StepFailed` — so the event is currently unreachable. No state in the parser for "awaiting user choice". |
| `src-tauri/src/runs/session.rs` | `input_rx` channel is a generic stdin write path. No step-failure-specific routing or state. Cancellation uses `child.kill()` with a 200 ms drain. |
| Epic T4.8 note | "If stdin tokens don't work, Continue = re-invoke the step. Retry = same. Skip = advance to next step. Abort = kill and mark failed." — confirms (b) was anticipated. |

### Chosen protocol

**(b) kill + re-invoke**, per-choice plan (now in KB §9 item 3 and `patterns/step-failure-protocol.md`):

- **Continue**: write `"\n"` to stdin; kill + re-invoke after 2 s if child stalls. Auto at 60 s timeout.
- **Retry**: `child.kill()` + re-invoke same `LaunchInput`; new `run_id`; `meta.json` records `retry_of`.
- **Skip**: `child.kill()` + re-invoke with `"Skip the previous failing step and continue."` prefix.
- **Abort**: `child.kill()` + mark `RunStatus::Failed`, `exit_note = "Aborted by user"`.
