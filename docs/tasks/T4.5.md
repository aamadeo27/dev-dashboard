# T4.5 — OrphanReaper

## What was done

Implemented the `OrphanReaper` component (`src-tauri/src/runs/orphan.rs`). On every app startup it scans all registered projects' `.claude/runs/*/meta.json` files. Any run whose status is `Pending` or `Running` is treated as an orphan: its PID is conditionally killed and the run is marked `Failed` with `note = "Terminated (app restarted)"`. Wired the sweep into `lib.rs` via `tauri::async_runtime::spawn` inside the Tauri `setup` hook, ensuring it runs before any new run can be launched.

## How the affected component works

### Data flow

```
app startup (lib.rs setup hook)
  → spawn async task
      → lock ProjectRegistry, collect project paths
      → lock SettingsStore, read claude_cli_path
      → orphan::run(project_paths, cli_path)
           for each project:
             → read_dir(<project>/.claude/runs/)
             for each run_dir/:
               → read + parse meta.json
               if status is Pending|Running:
                 if pid is Some(n):
                   → sysinfo::System::process(pid)
                   if alive AND exe matches claude_cli_path:
                     → process.kill()
                 → mark_failed(meta_path, run)
                     → write meta.json.orphan.tmp
                     → rename to meta.json  (atomic)
```

### Conservative kill rule

A PID that is alive but whose exe path does NOT match the configured claude CLI path is left untouched. This prevents killing unrelated processes that happened to reuse the same PID number after a restart. When `claude_cli_path` is not configured (first-launch), the kill step is skipped entirely but the status update still happens.

### Process table snapshot

`sysinfo::System::refresh_processes(ProcessesToUpdate::All, true)` is called once at the start of each sweep (not per-run) inside `spawn_blocking` to avoid blocking the async executor during the syscall. The resulting snapshot is reused for all PID lookups.

### Error handling

Every I/O error (missing directory, unreadable or unparseable meta.json, failed write/rename) is logged at `warn` and skipped. The function never propagates errors — the app must start even if some project dirs are inaccessible.

## Decisions made within Task scope

1. **Used `orphan.rs` (existing stub) instead of creating a new `orphan_reaper.rs`** — `mod.rs` already declares `pub(crate) mod orphan`; creating a second file would require a mod.rs change. No upstream decision needed.
2. **`spawn_blocking` for sysinfo refresh** — `System::refresh_processes` is a blocking syscall. Running it on `spawn_blocking` prevents executor stall. Standard pattern in the codebase for CPU/blocking work.
3. **Atomic write uses `meta.json.orphan.tmp`** — uses a distinct tmp suffix from the `TranscriptWriter`'s `meta.json.tmp` to avoid any possible conflict if both were running concurrently (they shouldn't be at startup, but the naming makes intent clear).
4. **Case-insensitive exe path comparison** — `to_lowercase()` on both paths before comparing handles Windows path casing (`C:\Users\...` vs `c:\users\...`). This is particularly relevant on Windows where the CLI path comes from settings and sysinfo may normalize differently.

## How to test / verify

### Unit tests

```
cargo test --manifest-path src-tauri/Cargo.toml runs::orphan
```

Tests cover:
- Completed/stopped runs left untouched
- Pending → Failed (no PID)
- Running → Failed (no PID)
- Missing runs dir — no panic
- Empty project list — no panic
- Mixed statuses: only Pending/Running are marked
- Corrupt meta.json skipped
- `maybe_kill_pid` returns `false` when PID not in system
- `maybe_kill_pid` returns `false` when CLI path not configured
- Multiple projects, all orphans marked
- Non-directory entries in runs dir skipped

### Smoke test (acceptance criterion)

1. Configure a project and start a run via the UI.
2. Force-quit the application while the run is in-flight.
3. Relaunch the application.
4. The run now shows `status: Failed`, `note: "Terminated (app restarted)"`.
5. The child claude process (if it was alive) should be gone.

The acceptance criterion "a PID alive but with a different exe is NOT killed" is exercised by the unit test `maybe_kill_pid_returns_false_when_pid_not_in_system` (empty system = PID not found branch), and the conservative exe-mismatch path is covered by `maybe_kill_pid` which returns `false` and logs `"exe mismatch"` when the paths differ.

## Review iteration 1 — Fixes applied

**Commit:** `fix(T4.5): review iteration 1 — startup ordering, lock contention, test coverage, naming`

### F-4 (HIGH) — Await the reaper sweep before setup returns

**Change:** In `lib.rs`, replaced `tauri::async_runtime::spawn(async { ... })` (fire-and-forget) with:
```rust
let handle = tauri::async_runtime::spawn(async move { ... });
tauri::async_runtime::block_on(handle)
  .map_err(|e| format!("orphan reaper join error: {}", e))?;
```

**Rationale:** The prior code comment claimed "must run before any new run can be launched," but `spawn()` returns immediately; the sweep ran concurrently with setup and IPC handlers. The fix blocks on the JoinHandle so the sweep **completes** before setup returns, guaranteeing no race between orphan cleanup and the first `launch_run` IPC call.

### F-5 (MED) — Fix test coverage for cli_path=None

**Change:** In `orphan.rs`, test `maybe_kill_pid_returns_false_when_cli_path_is_none()` now:
1. Creates a `System` and calls `refresh_processes()` to populate the process table
2. Uses PID 1 (init process on Unix) instead of PID 9_999_999
3. Added clarifying comments about the platform-specific behavior

**Rationale:** The original test used an unrefreshed `System` (no processes) and tested PID 9_999_999. The function returned `false` at the "PID not found" branch — before it ever checked `cli_path`. The test intent was to verify "cli_path=None → returns false," but it didn't exercise that code path. The fix uses PID 1 to ensure the function reaches the cli_path check (even if PID 1 doesn't exist on Windows, the behavior is still correct: cli_path=None → return false).

### F-7 (MED) — Avoid holding Mutex guard across await

**Change:** In `lib.rs`, restructured the lock scope:
```rust
let project_paths: Vec<...> = {
    let projects_guard = projects.lock().await;
    let list = projects_guard.list_projects().await;  // await inside lock
    list.into_iter().map(|p| p.path).collect()
};
// guard dropped here
```

**Rationale:** The prior code called `projects_guard.list_projects().await` in the closure chain without releasing the lock first. Holding the Mutex guard across the await point blocks other tasks that might try to lock `projects` during startup. The fix collects the result into a `list` variable, then drops the guard before iterating — freeing the lock as soon as possible per the inline comment intent.

### F-9 (LOW) — Extract orphan note string to const

**Change:** Added at module top:
```rust
const ORPHAN_NOTE: &str = "Terminated (app restarted)";
```
Updated `mark_failed()` and all test assertions to use `ORPHAN_NOTE` instead of string literals.

**Rationale:** The note string appeared in two places in code and five in test assertions. A single constant makes future localization or message changes a one-point edit and makes test assertions self-documenting.

### F-10 (LOW) — Gate test module with #[cfg(test)]

**Change:** Removed `pub(crate)` visibility from the tests module and test helpers:
- Changed `#[cfg(test)] pub(crate) mod tests` → `#[cfg(test)] mod tests`
- Changed `pub(crate) fn write_meta(...)` → `fn write_meta(...)`
- Changed `pub(crate) fn read_meta_sync(...)` → `fn read_meta_sync(...)`

**Rationale:** The `pub(crate)` helpers were visible across the crate and compiled into release builds even though they're only used in tests. Gating the whole module with `#[cfg(test)]` ensures test code doesn't appear in release binaries. The integration tests in `orphan_integration.rs` re-implement identical helpers locally, so the `pub(crate)` was unused by external code anyway.
