# T4.3 — Findings: Review Iteration 1

Four reviewers ran in parallel against the T4.3 codebase (RunManager, RunSession, launch_run / stop_run / send_input commands).

## CRITICAL (Architect escalation)

### C1 — CLI invocation argument strategy (Scope reviewer)
**File:** `src-tauri/src/ipc/commands.rs:618`

Current code passes `sequence_name` as the sole positional argument to the claude CLI:
```rust
cmd.arg(&input.sequence_name)
```
This is an unreviewed design decision. The correct invocation strategy (positional arg vs `--print`, vs piping sequence content on stdin, vs `--resume`, etc.) is an Architect-level choice with security and UX implications. The fix-pass coder must not change CLI invocation shape until the Architect has responded.

**Escalated to:** `evo_architect`

## HIGH

### H1 — `sequence_name` not validated (Security reviewer)
**File:** `src-tauri/src/ipc/commands.rs:618`

`sequence_name` is passed directly to `cmd.arg()` with no validation. A value starting with `-` (e.g., `--print`, `--help`, `--version`) would be interpreted as a CLI flag, not a sequence name. This is argument injection.

Fix: reject any `sequence_name` that is empty, starts with `-`, or contains characters outside `[A-Za-z0-9._\- /]`. Apply in `launch_run` before the spawn step.

---

### H2 — `send_input` text has no length cap (Security reviewer)
**File:** `src-tauri/src/ipc/commands.rs:741`

`text` is written verbatim to child stdin with no size limit. A frontend caller (or malicious IPC client) can write unbounded bytes, exhausting the stdin pipe buffer or causing OOM.

Fix: reject `text.len() > 65_536` (64 KiB) with `AppError::InvalidInput`.

---

### H3 — Drain skipped when cancelled (Scope reviewer)
**File:** `src-tauri/src/runs/session.rs:213`

```rust
if !cancelled {
    // drain stdout + stderr
}
```
After `child.kill()` the child may still flush final bytes before the pipe closes. Skipping the drain loses those events (e.g., a final error message). The drain should always run — the 200ms timeout already bounds it.

Fix: remove the `if !cancelled` guard; always enter the drain block.

---

### H4 — UserInput not written to transcript (Scope reviewer)
**File:** `src-tauri/src/ipc/commands.rs:714`, `src-tauri/src/runs/session.rs`

`send_input` writes text to child stdin but emits no `UserInput` event to the transcript. The `TranscriptWriter` is exclusively owned by `run_io_loop`, so `send_input` cannot call `writer.append_event` directly.

Fix: add an `input_tx: tokio::sync::mpsc::UnboundedSender<String>` field to `SessionHandle`. `send_input` sends the text via this channel. `run_io_loop` selects on `rx.recv()` alongside stdout/stderr, writes a `RunEvent::UserInput` to transcript, and emits `run:event` to the frontend.

---

### H5 — `span.enter()` not async-aware (Quality reviewer)
**File:** `src-tauri/src/runs/session.rs:106-107`

```rust
let span = tracing::info_span!("run_session", run_id = %run_id);
let _enter = span.enter();
```
`span.enter()` returns a synchronous guard that drops at the first `.await`. Tracing context is missing for all async work after the first yield point.

Fix: replace with `.instrument(span)` at the `tokio::task::spawn` call site in `commands.rs:655`, or use `span.entered()` inside the async block.

---

### H6 — Per-event `emit()` in hot I/O loop (Performance reviewer)
**File:** `src-tauri/src/runs/session.rs:163-171`

One `serde_json::json!` heap allocation plus one `app_handle.emit()` IPC call is issued per `RunEvent` in the hot stdout loop. At high event rates (1,000+/sec) this is significant overhead.

Fix: collect all events from a single `read()` call into a `Vec`, then emit one batched payload per `read()`:
```rust
let payload = serde_json::json!({ "run_id": run_id, "events": events_parsed });
app_handle.emit(events::RUN_EVENT, payload)
```
(Frontend listener must be updated to handle `events` array instead of single `event`.)

## MEDIUM

### M1 — Double-lock pattern repeated three times with no abstraction (Perf + Quality reviewers)
**Files:** `commands.rs:644-647`, `commands.rs:688-691`, `commands.rs:719-722`

```rust
let sessions = {
    let rm = state.run_manager.lock().await;
    rm.sessions.clone()
};
```
Appears identically in `launch_run`, `stop_run`, and `send_input`. The intent (avoid holding `run_manager` lock during session work) is non-obvious.

Fix: add `pub fn sessions_arc(&self) -> Arc<Mutex<HashMap<String, Arc<SessionHandle>>>>` on `RunManager`.

---

### M2 — Sequential drain uses two separate timeouts (Performance reviewer)
**File:** `src-tauri/src/runs/session.rs:217-247`

Stdout and stderr are drained sequentially with separate 200ms timeouts, for a worst-case 400ms blocking drain.

Fix: use `tokio::join!` to drain both concurrently under a single shared 200ms timeout.

---

### M3 — Windows path prefix check uses string comparison (Security reviewer)
**File:** `src-tauri/src/runs/session.rs:71`

`run_dir_norm.starts_with(&prefix_norm)` is a string prefix check. A `run_dir` of `.claude/runs-evil/abc` would pass if the prefix is `.claude/runs` (no trailing separator). On Windows with 8.3 aliases the canonicalization may also produce different casing.

Fix: add a trailing `/` (or `\\`) to `prefix_norm` before the `starts_with` check, so `.claude/runs/` cannot be confused with `.claude/runs-evil/`.

---

### M4 — Session inserted into map before I/O task is spawned (Security reviewer)
**File:** `src-tauri/src/ipc/commands.rs:649-665`

The session handle is inserted into the sessions map at line 651, but the I/O task is not spawned until line 655. In the window between these two lines, `stop_run` could acquire the handle and fire the cancellation token before the task has started.

Fix: spawn the I/O task first, then insert the handle. The cancellation token fires independently; the task will pick it up on its first `select!` iteration.

---

### M5 — Drain logic duplicates the main I/O loop body (Quality reviewer)
**File:** `src-tauri/src/runs/session.rs:217-247`

The drain block repeats ~20 lines from the main `select!` stdout arm (feed → parse → emit). These will diverge silently if the main loop changes.

Fix: extract a `process_stdout_chunk` helper and call it from both the main loop and the drain block.

---

### M6 — `#[allow(clippy::too_many_arguments)]` on a 9-parameter public function (Quality reviewer)
**File:** `src-tauri/src/runs/session.rs:94`

Nine parameters is a design signal. `stdout`, `stderr`, `child`, `writer`, `cancel`, `run`, and `sessions` are all per-run I/O state.

Fix: introduce a `RunIoContext` struct grouping these fields; `run_io_loop` takes `(app_handle, run_id, ctx: RunIoContext)`. Can be done as part of the H4 (UserInput channel) rework since that adds an `input_rx` field anyway.

### L2 — Unnecessary intermediate binding for `exit_code` (Quality reviewer)
**File:** `src-tauri/src/runs/session.rs:253-255`

```rust
let exit_code: Option<i32> = exit_status
    .ok()
    .and_then(|s| s.code());
```
`exit_status` is used only once; the intermediate variable adds noise.

### L3 — `list_runs` / `get_run` absent without documented deferral (Scope reviewer)

KB §5.3 lists `list_runs` and `get_run` as run commands. Neither is implemented. This should be explicitly noted as deferred in T4.3.md.

---

### L4 — `log_frontend_error` uses UUID v4, not v7 (minor consistency)
**File:** `src-tauri/src/ipc/commands.rs:267`

All other ID generation in this project uses `Uuid::now_v7()`. The correlation_id here uses `Uuid::new_v4()`. Low impact but inconsistent.
