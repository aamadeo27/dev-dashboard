# T0.6 — Logging and Tracing: `log_frontend_error` IPC command

## What was done

Added the `log_frontend_error` Tauri IPC command so the React frontend can forward
unhandled errors (e.g. from an `ErrorBoundary`) to the backend's tracing subscriber
and into the rotating log file already configured by T0.2.

- Implemented the command in `src-tauri/src/ipc/commands.rs`
- Registered it in `src-tauri/src/lib.rs` `invoke_handler!`
- Added a typed TypeScript wrapper in `src/ipc/commands.ts`
- Wrote Rust integration tests in `src-tauri/tests/log_frontend_error.rs`
- Wrote frontend Vitest tests in `src/ipc/log_frontend_error.test.ts`

`src-tauri/src/logging.rs` was **not modified** — it was already complete.

---

## How the affected component works

### Backend (`log_frontend_error` command)

The Tauri command receives three fields from the IPC call:

| Field     | Type            | Description                          |
|-----------|-----------------|--------------------------------------|
| `message` | `String`        | Error message text (required)        |
| `stack`   | `Option<String>`| Stack trace (optional)               |
| `route`   | `Option<String>`| Frontend route at time of error      |

It calls `tracing::error!` with structured fields:

```
component = "frontend"
message   = <value>
stack     = <value or "">
route     = <value or "">
```

The log line is written at `error` level and routed through the existing
`tracing-subscriber` registry initialised by `logging::init_logging()`, which
writes JSON-formatted lines to `<config_dir>/logs/app-YYYY-MM-DD.log`.

The function has no return value — fire-and-forget. If the non-blocking
writer is saturated it silently drops the line; the frontend never sees an
error.

### Frontend wrapper (`logFrontendError`)

```typescript
import { logFrontendError } from "@/ipc/commands";

await logFrontendError(
  "TypeError: Cannot read properties of undefined",
  error.stack,       // optional
  location.pathname, // optional
);
```

The wrapper calls `invoke("log_frontend_error", { message, stack, route })`.
Optional args are passed as `undefined` when omitted; Tauri's JSON deserialiser
maps JSON `null`/absent to Rust `None`.

Typical usage site: `src/components/common/ErrorBoundary.tsx` in
`componentDidCatch` or the `onError` callback of a React Query boundary.
Full IPC command wrappers for other commands are deferred to **T0.4**.

## Decisions made within Task scope

- Used `stack.as_deref().unwrap_or("")` rather than `Option`-typed tracing fields
  because an empty string is a cleaner JSON value than a missing/null field when
  parsed by log aggregators.
- Rust test file uses `#[tokio::test]` (matching the async command signature) with
  no tracing subscriber initialised — the `tracing` macros are no-ops when no
  subscriber is registered, so the tests never panic.
- Frontend tests mock `@tauri-apps/api/core` at module level so no Tauri runtime
  is required. The mock is registered with `vi.mock` before the import.

## How to test / verify

### Frontend (runnable without Rust)

```sh
pnpm lint:ci    # biome formatting + lint — must exit 0
pnpm typecheck  # tsc --noEmit — must exit 0
pnpm test       # vitest run — 5 tests in log_frontend_error.test.ts must pass
```

### Backend (requires Rust toolchain)

```sh
cargo test --manifest-path src-tauri/Cargo.toml --test log_frontend_error
```

This runs the three no-panic tests in `src-tauri/tests/log_frontend_error.rs`.

### Manual end-to-end

1. Run `pnpm dev:local` to start the app with `DEV_DASHBOARD_LOG=debug`.
2. From browser DevTools console: `window.__TAURI__.core.invoke("log_frontend_error", { message: "test", stack: "trace", route: "/dashboard" })`.
3. Open `.dev-data/logs/app-YYYY-MM-DD.log` and confirm a JSON line with
   `"level":"ERROR"`, `"component":"frontend"`, and the correct field values.
